# 评测与基准：accuracy/latency/throughput

## 要点

- 评测需要同时覆盖：正确性（或质量）与性能（吞吐、延迟、尾延迟）
- 基准要可复现：固定硬件/软件版本、固定输入集、明确 warmup 与统计方法
- 评测的核心不是“数字越多越好看”，而是要回答：**模型是否更强、系统是否还能跑、成本是否可接受**。

## 通用知识

### 它是什么

评测与 benchmark 是把模型或系统的表现量化成一组可复现指标，用于：

- 比较不同版本
- 定位回退
- 支撑上线决策

### 它解决什么问题

它要回答至少三类问题：

- 质量有没有变好
- 性能有没有变差
- 成本和稳定性是否仍可接受

### 为什么在 AI 系统里重要

因为大模型改动往往会牵一发动全身：

- 量化可能提升吞吐但伤精度
- 新 kernel 可能加速平均值却拉高 p99
- 对齐或后训练可能提升行为指标但伤基础能力

### 它的收益与代价

收益：

- 让优化和回归有证据
- 降低“拍脑袋上线”风险

代价：

- 需要维护基准集、环境、统计方法
- 指标设计不当时，可能会把团队带偏

## 指标

- 质量：任务集得分、人工评审、回归样例
- 性能：
  - throughput：tokens/s、req/s
  - latency：TTFT（首 token 时间）、TPOT（每 token 时间）、p50/p95/p99
  - 资源：显存占用、GPU 利用率、CPU 利用率

补一个很实用的分层方法：

1. 能力指标：会不会做题
2. 系统指标：能不能稳定跑起来
3. 成本指标：代价是否可接受
4. 行为指标：输出是否符合产品要求

## 基准设计

- 固定输入长度分桶：短/中/长
- 区分 prefill 与 decode
- 并发水平：1、N（接近线上）

## 最小例子

假设你把模型从 FP16 切到 W8A16：

- 平均 tokens/s 提升 20%
- p95 TTFT 恶化 8%
- 某些任务 benchmark 下降 2 个点

这说明“量化更快”这个结论本身不完整。

更准确的说法应该是：

- 吞吐提升了
- 首 token 体验未必更好
- 某些能力可能有回退

这正是 benchmark 需要多维汇报的原因。

## 工程例子

一个常见误判是：

- 只在单请求离线环境里测到性能提升
- 然后直接假设线上并发环境也会同步提升

但真实线上还会引入：

- 排队
- 动态 batching
- KV cache 竞争
- allocator 碎片
- 长短请求混跑

所以离线 benchmark 和线上表现经常不是一回事。

## 推理优化工程师视角

从推理优化的角度，这一章最重要的作用是帮你避免“只优化自己想看的指标”。

- kernel 工程师容易只看单算子耗时
- serving 工程师容易只看 tokens/s
- 模型工程师容易只看任务集分数

但真正能指导上线决策的，通常是一组联合指标：

- 质量是否守住底线
- TTFT / TPOT / p99 是否可接受
- 显存、GPU 利用率、并发能力是否匹配目标成本

所以做优化时，最好先定义清楚“这次优化到底为谁服务”：

1. 是为了提升单机吞吐？
2. 是为了压低首 token 延迟？
3. 是为了降低显存，让更大 batch 或更长上下文可运行？

没有这个目标约束，benchmark 很容易变成漂亮但无决策价值的数字表。对推理优化工程师来说，评测体系本身就是优化方法论的一部分。

## 常见面试问题

### 初级

1. throughput、TTFT、TPOT 分别描述什么？
2. 为什么 benchmark 需要固定硬件、软件版本和 warmup 策略？

### 中级

1. 为什么只看平均延迟不够？
2. 为什么离线 benchmark 好，不代表线上体验一定好？

### 高级

1. 如果一个优化让 tokens/s 提升，但 p99 恶化，你如何判断是否值得上线？
2. 如果两个版本质量接近，但一个显存更低、尾延迟更稳，你会怎么做决策？

## 易错点

- 只做单条请求 benchmark，和线上差距巨大
- 忽略 warmup/缓存（CUDA graph、kernel cache、编译 cache）
- 只追单一 benchmark 分数，忽略了真实 workload 和行为约束
- 把训练指标、能力指标和系统指标混成一个“总分”

## 排查 checklist

- [ ] benchmark 脚本是否固定了版本与参数？
- [ ] 是否按长度分桶报告 p95/p99？
- [ ] 是否记录了运行时的 GPU/CPU/显存曲线？
