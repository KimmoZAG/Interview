# Reward 与 Verifier 设计：后训练到底在优化什么

## 要点

- 后训练真正难点是监督信号可靠性，而不是算法缩写本身。
- reward 回答“鼓励什么输出”，verifier 回答“是否满足可检查标准”。
- 设计不好会诱发 reward hacking。

## 通用知识

### 它是什么

在后训练里，模型不会凭空知道“什么叫更好的回答”，必须通过某种监督信号去学习。

最常见的两类信号是：

- reward：给回答一个分数、偏好或排序，表达“更想鼓励哪种输出”
- verifier：检查输出是否满足某个可验证标准，例如答案是否正确、格式是否合规、代码测试是否通过

### 它解决什么问题

它们解决的是：

- 训练目标不再只是“拟合下一个 token”
- 而是让模型朝着“更像人想要的结果”优化

如果没有这层信号设计，后训练就很容易停留在“看起来像学了，但不知道到底被鼓励了什么”。

### 为什么在工程里重要

因为后训练成败往往不只取决于算法名字，而更取决于：

- 信号本身靠不靠谱
- 信号和产品目标是不是一致
- 模型是否学会了“拿高分的捷径”而不是真变好

## Reward vs Verifier

- Reward：给分或偏好，驱动策略优化
- Verifier：可检查约束（答案正确、格式合法、测试通过等）

## 最小例子

假设你在做一道数学题生成任务：

- reward 可能来自人工偏好或模型打分，鼓励“看起来更像好答案”的输出
- verifier 可能直接检查最终答案是否正确，或者步骤是否满足某些约束

这两类信号都可能有价值，但风险不同：

- reward 更灵活，却可能被模型学会迎合表面偏好
- verifier 更刚性，却只适合那些“能被自动检查”的任务

所以实际系统里，常常不是二选一，而是要弄清楚它们各自能覆盖什么、会漏掉什么。

## 最小设计框架

1. 明确要鼓励的行为
2. 判断是否可自动检查
3. 预判模型可能的投机路径
4. 校验与产品目标一致性
5. 验证副作用（能力、成本、稳定性）

## 工程例子

一个常见问题是：

- 某个 reward 模型把“长、像样、解释多”的回答打得更高
- 结果策略模型学会了生成更长、更保守、看起来更体面但不一定更有用的回答

从离线分数看，系统似乎进步了；但上线后可能出现：

- 回复变啰嗦
- 成本上升
- 用户并不觉得更好

这就是典型的 reward misalignment：优化目标和真实目标之间有缝。

同样地，verifier 也不是万能的：

- 如果 verifier 只检查最终答案，模型可能学会投机式构造
- 如果 verifier 检查规则过窄，模型可能只会讨好规则，而不是提升真实能力

## 常见面试问题

### 初级

1. reward 和 verifier 的区别是什么？
2. 为什么说后训练难点常在信号设计，而不只是算法选择？

### 中级

1. 什么是 reward hacking？
2. 为什么 verifier 在某些任务上很好用，但不是通用解法？

### 高级

1. 如果离线 reward 分数提升了，但用户体验没变好，你会如何排查？
2. 你会如何设计一套既能自动评估、又不容易被模型钻空子的 reward/verifier 体系？

## 为什么放在附录

这篇和后训练、对齐、产品指标都强相关，但它不是推理优化主线里的基础组件。

放在附录更合适，因为：

- 它帮助你理解后训练到底在“优化什么”
- 它适合作为面试中的延伸追问素材
- 它能把训练信号、评测信号和产品目标三者串起来

## 易错点

- 背 DPO/PPO/GRPO 名字，不清楚信号来源
- 把 verifier 当万能真理机
- 只看单指标，不做对抗样本检查

## 排查 checklist

- [ ] 当前信号具体在鼓励什么行为？
- [ ] 是否存在可利用漏洞？
- [ ] 与产品目标一致性如何？
- [ ] 是否有独立评测证明“真的变好”？
