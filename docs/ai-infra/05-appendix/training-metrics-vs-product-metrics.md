# 训练指标 vs 产品指标：为什么 loss 变好不代表产品更好

## 要点

- 训练指标、能力指标、系统指标、产品指标回答的是不同问题。
- validation loss 改善不等于用户体验、成本和稳定性同步改善。

## 通用知识

### 它是什么

这篇讨论的是一件在大模型项目里极其常见、也极其容易吵起来的事：

- 模型团队说“loss 变好了”
- 产品团队说“用户体验没变好”
- 系统团队说“成本和延迟还变差了”

问题通常不在于谁错了，而在于大家看的不是同一层指标。

### 它解决什么问题

它帮助你把一次改动的影响拆成不同层次：

- 训练是否更收敛
- 能力是否更强
- 系统是否更稳更便宜
- 产品是否真的更可用

### 为什么在工程里重要

因为大模型项目很少只追一个目标。

一个改动可能同时带来：

- loss 更低
- 某些 benchmark 更高
- latency 更差
- 用户满意度没明显提升

如果没有分层指标框架，团队就很容易各讲各话。

## 四层指标框架

1. 训练层：loss / perplexity
2. 能力层：benchmark accuracy / task success
3. 系统层：TTFT / TPOT / p95/p99 / tokens/s
4. 产品层：满意度、格式遵循率、违规率、完成率

## 最小例子

假设你做了一个后训练或蒸馏版本更新，结果是：

- validation loss 更低
- 通用 benchmark 提升 1 分
- 线上 TTFT 变慢 15%
- 用户主观反馈几乎没变

这时不能简单说“模型变好了”或“模型没变好”。

更准确的表达应该是：

- 训练层和能力层有提升
- 系统层有明显代价
- 产品层收益尚不明显

这就是为什么单看一个指标几乎总会得出片面的结论。

## 常见错配

- loss 下降但指令遵循变差
- benchmark 提升但延迟与成本恶化
- 对齐指标提升但基础能力回退

## 工程例子

一个典型项目复盘会遇到这样的局面：

- 模型团队推动上线，因为离线 benchmark 全面变好
- 线上团队反对上线，因为 p99 和单位成本明显恶化

如果此时没有提前定义“上线判定优先看哪几层指标”，讨论很容易卡死。

更成熟的做法通常是：

1. 先明确底线指标：例如延迟、违规率、关键任务成功率
2. 再定义加分指标：例如更高的 benchmark 或更好的用户偏好
3. 最后用 A/B 或 shadow eval 看产品层是否真的买单

这样做的好处是：即便团队目标不同，也能在同一张指标表上讨论。

## 常见面试问题

### 初级

1. 为什么 loss 变好不一定代表产品变好？
2. 系统指标和产品指标分别关注什么？

### 中级

1. 为什么 benchmark 提升不一定值得上线？
2. 如何向非模型同学解释“训练指标好”和“用户体验好”不是一回事？

### 高级

1. 如果离线质量提升，但成本和尾延迟恶化，你会如何做上线决策？
2. 你会如何设计一套能够同时服务模型、系统和产品团队的分层指标框架？

## 为什么放在附录

这篇不是推理优化主线里的基础概念，但它很适合作为“项目决策补丁”：

- 帮你把训练、能力、系统、产品四层指标串起来
- 帮你在面试或复盘里解释为什么不同团队会得出不同结论
- 帮你避免被单一指标牵着跑

## 排查 checklist

- [ ] 这次改动改善的是哪一层指标？
- [ ] 是否存在某层变好、某层变差？
- [ ] 是否同时报告了性能与成本？
- [ ] 离线评测分布与线上流量差异多大？
