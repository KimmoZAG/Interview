# 量化基础（INT8/INT4）与误差

## 要点

- 量化的工程目标通常是：更低显存、更高吞吐、更低成本（但需要验证精度）
- 关键决策：量化粒度（per-tensor / per-channel / group-wise）、校准方式、激活是否量化
- 量化不是“把 bit 数改小”这么简单，而是在 **速度 / 显存 / 精度 / 工程复杂度** 之间做交换

## 通用知识

### 它是什么

量化的核心是把原本高精度数值映射到更低 bit 宽的表示，例如 FP16/BF16 进一步压到 INT8、INT4。

常见目标包括：

- 权重更省显存
- 访存带宽压力更低
- 在支持低精度计算的硬件上提高吞吐

### 它解决什么问题

在 LLM 推理里，很多瓶颈并不只是算力不够，而是：

- 模型权重太大，装不下或加载慢
- decode 阶段反复读权重/KV，带宽吃紧
- 大 batch 或高并发时显存紧张

量化就是在减少每个元素占用字节数，从而降低这些压力。

### 它的收益与代价

收益：

- 更低显存占用
- 更低带宽消耗
- 某些场景下更高 tokens/s

代价：

- 误差引入可能损伤精度
- 校准和部署链路更复杂
- 不同硬件/框架对不同量化格式支持不一致

## 量化的最小概念

- scale/zero-point（仿射映射）
- 对称 vs 非对称量化
- 权重量化 vs 激活量化

## LLM 推理里的常见点

- 权重量化（W8A16、W4A16、W4A8 等组合）
- KV cache 的精度与存储（可能影响吞吐与质量）

补一个很实用的工程直觉：

- prefill 阶段往往更容易吃到大矩阵乘的低精度收益
- decode 阶段很多时候收益不只来自低精度算力，还来自更低带宽和更小 cache 占用

## 最小例子

假设一个权重矩阵有 $10^9$ 个元素：

- FP16 存储约需要 2GB
- INT8 存储约需要 1GB
- INT4 存储约需要 0.5GB（不含额外 scale / packing 开销）

这就是为什么量化首先经常被当成“显存手段”。

但注意：

- 真正部署时还会有 scale、group metadata、packing 对齐和 runtime buffer 开销
- 所以实际节省通常不会像纸面公式那样完美

## 工程例子

一个很典型的线上现象是：

- 把模型从 FP16 换成 W8A16 后，吞吐提升明显
- 但 p99 没有同步改善，甚至某些请求质量明显变差

可能原因包括：

- 长尾请求仍受调度或 KV cache 限制，不是单纯权重带宽问题
- 某些层对量化更敏感，导致生成质量波动
- benchmark 只看平均 tokens/s，没有分桶看长短请求

## 推理优化工程师视角

对推理优化工程师来说，量化最重要的不是背格式名词，而是形成三个判断：

1. 当前瓶颈到底更像算力、带宽，还是显存
2. 量化的收益会主要体现在 prefill、decode，还是模型装载与并发能力上
3. 所谓“更快”是否同时经过了质量、p95/p99 和显存曲线验证

换句话说：

- 量化不是默认提速按钮
- 它更像一个在 **模型质量、系统成本、部署复杂度** 之间做交换的系统动作

## 误差与验证

- 先选一个可复现评测（perplexity/任务集/回归样例）
- 再看线上指标（答案质量、拒答率、毒性过滤等——取决于业务）

## 常见面试问题

### 初级

1. 对称量化和非对称量化有什么区别？
2. 为什么量化能降低显存和带宽压力？

### 中级

1. per-tensor、per-channel、group-wise 量化分别适合什么场景？
2. 为什么 W4A16 和 W8A8 的工程取舍可能不同？

### 高级

1. 为什么量化后平均吞吐上升了，但 p99 不一定改善？
2. 如果某模型量化后数学和代码能力下降明显，你会优先排查什么？

## 易错点

- 同一套量化参数对不同分布/不同 batch 可能不稳
- 只看吞吐不看尾延迟（p95/p99）
- 把纸面 bit 宽节省直接当成真实部署收益，忽略 metadata 和 kernel 支持
- 不区分是 compute-bound 场景还是 bandwidth-bound 场景，导致错误预期量化收益

## 排查 checklist

- [ ] 量化配置写清楚：W?A?、粒度、group size、校准集
- [ ] 性能数据按 shape/批大小分桶统计
- [ ] 精度回归是否可复现（固定随机种子/固定采样策略）
- [ ] 量化后的瓶颈是否真的从带宽/显存侧得到缓解？
