# 通信基础：latency / bandwidth / topology

## 要点

- 通信问题要先分：延迟受限还是带宽受限。
- 同样的 collective，在不同拓扑（PCIe/NVLink/IB）下表现差异很大。
- 多卡扩展效率下降，通常不是单点 bug，而是通信与同步链路累积所致。

## 通用知识

### 它是什么

做多卡系统时，通信性能至少要用 3 个词描述：

- latency：一条消息从发起到开始有效传输的固定开销
- bandwidth：链路单位时间能搬多少数据
- topology：设备之间怎么连接，决定数据走哪条路

### 它解决什么问题

这 3 个概念帮助你判断：

- 为什么小消息很多时，即便总数据量不大也会很慢
- 为什么大消息可能更接近带宽上限
- 为什么同样 8 卡，拓扑不一样，吞吐差很多

### 为什么在 AI 系统里重要

因为训练和推理里的 collective 并不只看总字节数：

- DP 的梯度同步可能更偏带宽问题
- 小 bucket 或频繁同步常常更偏 latency 问题
- TP / MoE / 跨机训练还会强依赖 topology

### 它的收益与代价

- 如果你能把问题分成 latency-bound 和 bandwidth-bound，优化动作会更准
- 如果这一步判断错了，就很容易出现“明明换了更快 kernel，整体还是不快”的尴尬局面

## 最小例子

考虑两种通信：

1. 传 1KB，传很多次
2. 传 100MB，传很少次

第一种通常更容易被 latency 主导，因为每次启动通信的固定开销占比高。

第二种更容易被 bandwidth 主导，因为真正瓶颈变成链路能否持续搬大块数据。

所以“总共传了多少数据”并不能单独决定快慢，**消息大小分布** 同样重要。

## 工程例子

假设你在做分布式训练，发现：

- 单步总通信量没有夸张
- 但 step time 里通信占比仍然很高

一个常见根因是：

- bucket 太碎
- 每层或每几个算子就同步一次
- 导致消息非常多，通信被 latency 开销切碎

这时最有效的优化可能不是换网络，而是：

- 合并 bucket
- 减少同步频次
- 提高 overlap 比例

## 推理优化工程师视角

对推理优化工程师来说，这篇最重要的价值是建立一个很硬的判断顺序：

- 不要一看到通信慢，就先怀疑库或驱动
- 先判断是小消息太多，还是大消息太重
- 再判断问题主要落在 latency、bandwidth，还是 topology

这个顺序看起来朴素，但非常能避免误判。很多所谓“系统吞吐上不去”的问题，最后并不是算子慢，而是：

- 调度把通信切碎了
- batching 方式导致同步频率太高
- 原本机内可接受的模式，一跨机就被 latency 放大

把通信问题先翻译成这三个基本维度，后续排查路径会清楚很多。

## 常见面试问题

### 初级

1. latency 和 bandwidth 的区别是什么？
2. 为什么通信问题不能只看总字节数？

### 中级

1. 什么情况下通信更像 latency-bound？什么情况下更像 bandwidth-bound？
2. 为什么 topology 会影响同一个 collective 的性能？

### 高级

1. 当 32 卡扩展效率下降时，你如何判断主要是 latency、bandwidth 还是 topology 问题？
2. 为什么某些 workload 在单机扩展正常，但一跨机就明显退化？

## 最小框架

- latency：消息固定开销
- bandwidth：单位时间传输量
- topology：设备互联结构决定路径与拥塞

## 易错点

- 把“通信量大”误认为唯一问题，忽略消息数量和同步频率
- 只看平均带宽，不看是否存在局部拥塞和慢 rank
- 不分机内和跨机路径，导致排查对象错位

## 排查 checklist

- [ ] 当前慢的是少量大消息，还是大量小消息？
- [ ] 当前瓶颈更像 latency、bandwidth，还是拓扑绕路？
- [ ] 慢 rank 是否固定出现在某些设备或节点上？
- [ ] 通信是否和计算成功 overlap 了？
