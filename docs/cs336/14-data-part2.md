## 先抓住这讲要点

- 数据过滤的统一视角是：从一个巨大原始集合 $R$ 中，筛出一个更接近目标分布 $T$ 的子集 $T'$。
- 大规模过滤不能只追求“最准”，还要追求**足够快、足够便宜、足够可扩展**，因此轻量方法在工业里长期有价值。
- 去重绝不是清洁细节，而是 web-scale 训练的基本卫生：**exact dedup 和 near dedup 都得做**。

## 过滤的统一视角：从原始集合到目标集合

这是这讲最重要的抽象。  
无论你是在做：

- 语言识别；
- 质量过滤；
- 数学数据筛选；
- 代码数据挑选；
- 安全内容剔除；

本质上都可以理解成：

> 从一个很大、很杂的原始集合 $R$ 中，选出更接近目标集合 $T$ 的子集 $T'$。

也就是说，过滤不是在问“这篇文档好不好”，而是在问：

> 这篇文档是否更接近我现在想让模型学习的分布？

这是一种非常重要的思维转换。

## 一个统一的过滤流程长什么样

大多数过滤流程都逃不过这三步：

1. **建模或打分**；
2. **给每篇文档一个分数或标签**；
3. **根据阈值、排序或采样策略决定保留什么**。

这个框架之所以强，是因为它几乎能统一描述所有过滤任务。  
区别只在于：

- 你用什么特征；
- 你训练什么打分器；
- 你怎么设阈值；
- 你是否要分层采样而不是硬阈值切断。

## 为什么 n-gram 和轻量模型仍然很有生命力

很多人一听“数据过滤”，第一反应是拿大模型做 judge。  
但在 web-scale 数据工程里，这通常太贵。

### 因为第一阶段的任务往往不是“最精准”，而是“先把明显垃圾快速筛掉”

对于十亿级文档集合，工业上最常见的思路是：

- 先用极快、极便宜的方法做粗筛；
- 再对剩下的数据用更贵的方法细筛。

这也是为什么下面这些方法一直很实用：

- n-gram 规则；
- FastText 语言分类；
- 轻量质量分类器；
- 简单统计特征（长度、重复度、字符比例等）。

它们不一定最“聪明”，但它们足够：

- 快；
- 稳；
- 好扩展；
- 成本可控。

在工业里，这四点常常比“实验室里高 0.3 分”更重要。

## 高质量过滤为什么常用 teacher-student 思路

当你真的想从“大致可用”进一步走向“更高质量目标分布”时，轻量规则常常不够了。  
这时会出现一个常见套路：

1. 用更强模型或更贵标注，构造高质量标签；
2. 再训练便宜 student 模型去大规模打分；
3. 用 student 在海量数据上执行过滤。

这类 teacher-student 或蒸馏式过滤，本质上是在做一件事：

> 用少量高质量判断，扩展出大规模低成本近似判断。

它非常符合现实工程需求。

## 去重为什么是 web-scale 的必修课

如果不过滤重复数据，模型会出现几个问题：

- 反复学习同样内容，浪费 token budget；
- 某些内容被不成比例地放大；
- benchmark 污染风险上升；
- 训练分布被模板页、镜像页、转载页带偏。

所以 dedup 不是“可做可不做”的清洁步骤，而是**训练分布控制的核心环节**。

## exact dedup 和 near dedup 为什么都必须做

### 1. exact dedup：完全相同的重复

这是最简单的一层。  
同样的文档、同样的正文、同样的 hash，直接去掉即可。

它很重要，但远远不够。

### 2. near dedup：轻微改写和模板重复

互联网真实重复远比“字节完全一样”复杂。  
常见情况包括：

- 改了标题但正文一样；
- 模板页里只换了少量字段；
- 多站转载；
- 伪原创改写；
- 重复免责声明、license 文本、大段页脚。

这些都不会被 exact dedup 捕获，却同样会污染训练分布。

这就是为什么 near dedup 更难，也更关键。

## 代码拆解：Bloom Filter 直觉

```python
class TinyBloom:
    def __init__(self, m=1024):
        self.bits = [0] * m
        self.m = m

    def add(self, x):
        self.bits[hash(x) % self.m] = 1

    def contains(self, x):
        return self.bits[hash(x) % self.m] == 1
```

真实 Bloom Filter 会用多个 hash，但这段代码已经足够说明直觉：

> 我不需要完整记住每个对象，只需要用很省空间的方式快速判断“它可能见过没有”。

它适合做大规模预过滤，因为：

- 内存省；
- 查询快；
- 很适合海量集合的“先挡一层”。

## 代码拆解：MinHash / LSH 的一句话理解

```python
# 文档 -> shingles -> sketch -> buckets -> verify
```

这行伪代码其实非常经典：

1. 先把文档切成 shingles；
2. 再用 sketch（如 MinHash）压缩相似性特征；
3. 用 LSH 把相似文档送进同一个候选桶；
4. 再在桶内做更精确验证。

它最大的意义是：

> 把原本近乎不可做的 $O(N^2)$ 两两比较，压缩成一个工业上能跑得动的候选检索问题。

这就是为什么 MinHash + LSH 在大规模 near dedup 中长期流行。

## 分布匹配为什么比“过滤高质量文本”更重要

这点很容易被误解。  
很多人觉得只要“留下高质量文本”就行。  
但训练一个强模型，真正重要的是：

- 你的分布是否匹配目标能力；
- 你的数据比例是否合理；
- 某些领域有没有被过度放大或压低。

换句话说，数据工程不是只做二元判断“好/坏”，而是在做：

> 分布重塑。

比如你可能需要：

- 多留代码，少留垃圾论坛；
- 多留高价值数学解题，少留模板 SEO 文；
- 某些语种按目标比例采样，而不是按原始网页比例采样。

这才是“更接近目标集合 $T$”的真实含义。

## 面试里可以怎么讲

如果面试官问：**“为什么数据过滤可以统一看成一个问题？”**

你可以答：

> 因为无论做语言识别、质量过滤还是领域选择，本质上都是从原始数据集合中筛出更接近目标训练分布的子集。差别主要在打分器、阈值策略和采样方式，而不是问题本质。

如果面试官问：**“为什么 exact dedup 不够？”**

可以答：

> 因为 web 数据里大量重复是轻微改写、模板复用、转载镜像和伪原创，这些不会被简单 hash 去重捕获，但同样会扭曲训练分布，所以 near dedup 也必须做。

## 面试题答题提示

### 1. 过滤问题最好讲成分布逼近

这比“删垃圾文本”更准确，因为很多数据处理并不是非黑即白，而是在做更接近目标训练分布的筛选。

### 2. 去重要分 exact 和 near

这两类问题的技术路径完全不同，前者更像哈希集合，后者才需要 MinHash、LSH 这类近似技术。

### 3. 过滤系统要讲成本约束

超大规模数据处理中，推理成本本身就是设计的一部分，所以轻量模型和分层打分往往更现实。
