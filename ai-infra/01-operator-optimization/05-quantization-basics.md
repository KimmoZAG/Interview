# 量化基础（INT8 / INT4）与误差

## 一句话先讲清

量化不是“把 bit 数改小”这么简单，而是在 **速度、显存、带宽、精度、工程复杂度** 之间做交换。

它不是默认提速按钮，而是一个必须带着证据上场的系统动作。

## 为什么值得单独学

- 在 LLM 推理里，瓶颈很多时候不是纯算力，而是显存、带宽和加载成本。
- 量化常常一边带来吞吐收益，一边引入精度和部署链路复杂度。
- “更快”如果没有和质量、p95/p99、显存曲线一起验证，往往结论不够硬。

## 量化到底在解决什么问题

量化的核心，是把高精度数值映射到更低 bit 宽的表示，例如从 FP16 / BF16 进一步压到 INT8、INT4。

它常见想解决的问题包括：

- 权重太大，装不下或加载太慢
- decode 阶段反复读权重 / KV，带宽吃紧
- 高并发或长上下文时显存紧张

所以量化的本质是：**减少每个元素占用的字节数，从而缓解显存和带宽压力。**

## 最小概念，一次讲清

| 概念 | 直觉 |
|---|---|
| Scale / Zero-point | 把实数映射到低 bit 表示的参数 |
| 对称量化 | 围绕 0 更对称，表达更简洁 |
| 非对称量化 | 更灵活，但实现复杂度更高 |
| 权重量化 | 先减权重存储和读取成本 |
| 激活量化 | 再进一步压带宽和计算，但更敏感 |
| 粒度 | per-tensor / per-channel / group-wise，对误差和实现都有影响 |

## 在 LLM 推理里，最常见的收益来源

- 权重量化：如 `W8A16`、`W4A16`、`W4A8`
- KV cache 量化：影响长上下文下的带宽与容量

一个很实用的工程直觉：

- prefill 阶段更容易吃到大矩阵乘的低精度收益
- decode 阶段很多时候收益不只来自低精度算力，还来自更低带宽和更小 cache 占用

## 最小例子：纸面节省和真实节省不是一回事

假设一个权重矩阵有 $10^9$ 个元素：

- FP16 存储约需要 2GB
- INT8 存储约需要 1GB
- INT4 存储约需要 0.5GB

这就是为什么量化首先经常被当成“显存手段”。

但真实部署时还会有：

- scale
- group metadata
- packing / alignment
- runtime buffer

所以实际节省通常不会像纸面公式那样完美。别被理想数学骗太狠，工程会记仇。

## W4A16、W8A8 这些组合为什么要分开看

| 组合 | 常见工程含义 |
|---|---|
| W8A16 | 更常见、更稳，先减权重带宽和显存 |
| W4A16 | 节省更激进，但误差和实现要求更高 |
| W8A8 | 权重和激活都压缩，收益更大但风险更高 |

真正要问的不是“哪种更先进”，而是：

- 当前瓶颈更像算力、带宽，还是显存？
- 当前硬件 / runtime 对哪种低精度支持更成熟？
- 当前业务能接受多大的精度代价？

## Troubleshooting：为什么平均吞吐上去了，p99 却没变好

| 现象 | 第一怀疑点 | 如何验证 |
|---|---|---|
| 平均 tokens/s 上升，p99 没改善 | 长尾仍被调度或 KV 限制 | 按长短请求分桶看 latency |
| 质量波动明显 | 某些层或某类请求对量化更敏感 | 固定评测集与回归样例 |
| 显存降了，但收益不稳定 | 瓶颈不只在权重读取 | 联合看 KV、allocator、batching |
| benchmark 漂亮，线上一般 | workload 分布与测试集不一致 | 对齐线上请求形态与采样参数 |

### 一个排障顺序

1. 先判断当前瓶颈更像算力、带宽还是显存。
2. 再看量化收益主要应该落在 prefill、decode，还是装载 / 并发能力上。
3. 然后同时验证质量、p95/p99 和显存曲线。
4. 最后再决定是继续压 bit，还是回退到更稳的方案。

## 推理优化工程师视角

量化最重要的不是背格式名词，而是形成三个判断：

1. 当前瓶颈到底更像算力、带宽，还是显存。
2. 量化的收益会主要体现在 prefill、decode，还是模型装载与并发能力上。
3. 所谓“更快”是否同时经过了质量、p95/p99 和显存曲线验证。

换句话说：量化不是一个默认提速按钮，而是一个系统级交换动作。

## 面试高频问法

### 初级

1. 对称量化和非对称量化有什么区别？
2. 为什么量化能降低显存和带宽压力？

### 中级

1. per-tensor、per-channel、group-wise 量化分别适合什么场景？
2. 为什么 `W4A16` 和 `W8A8` 的工程取舍可能很不同？

### 高级

1. 为什么量化后平均吞吐上升了，但 p99 不一定改善？
2. 如果某模型量化后数学和代码能力下降明显，你会优先排查什么？

## 易错点

- 同一套量化参数对不同分布 / 不同 batch 可能不稳
- 只看吞吐，不看尾延迟（p95 / p99）
- 把纸面 bit 宽节省直接当成真实部署收益，忽略 metadata 和 kernel 支持
- 不区分 compute-bound 和 bandwidth-bound 场景，导致错误预期量化收益

## 排查 checklist

- [ ] 量化配置是否写清楚：W?A?、粒度、group size、校准集？
- [ ] 性能数据是否按 shape / batch 大小分桶统计？
- [ ] 精度回归是否可复现（固定随机种子 / 采样策略）？
- [ ] 量化后的瓶颈是否真的从带宽 / 显存侧得到缓解？
