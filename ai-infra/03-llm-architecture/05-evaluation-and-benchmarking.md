# 评测与基准：accuracy / latency / throughput

## 一句话先讲清

评测与 benchmark 的核心不是做出一张很好看的表，而是回答三件事：**模型是否更强、系统是否还能稳定跑、成本是否仍然可接受。**

如果这三件事没有同时被量化，所谓“优化成功”通常都不太稳。

## 为什么值得单独学

- 量化、融合、新 runtime、后训练……每一种改动都可能同时改变质量、吞吐、尾延迟和显存。
- 只看单一指标，很容易把团队带去优化一个“看起来对、上线不一定香”的方向。
- 真正有决策价值的 benchmark，必须能复现，也必须能解释。

## 评测到底在回答什么

至少要回答下面三类问题：

1. **质量** 有没有变好或退化？
2. **性能** 有没有变快，且是不是快在正确的指标上？
3. **成本与稳定性** 是否仍然满足上线要求？

可以再进一步拆成四层：

| 层次 | 典型问题 |
|---|---|
| 能力指标 | 模型会不会做题，基础能力有没有掉 |
| 系统指标 | TTFT / TPOT / throughput / p99 是否稳定 |
| 成本指标 | 显存、GPU 利用率、并发能力是否可接受 |
| 行为指标 | 输出是否符合产品风格与安全要求 |

## 常见指标应该怎么记

| 类别 | 常见指标 | 它回答什么 |
|---|---|---|
| 质量 | 任务集得分、人工评审、回归样例 | 模型能力有没有守住底线 |
| 吞吐 | tokens/s、req/s | 单位时间能处理多少工作 |
| 延迟 | TTFT、TPOT、p50/p95/p99 | 用户体感和尾延迟是否可接受 |
| 资源 | 显存、GPU/CPU 利用率 | 成本和部署约束是否满足 |

一句很实用的判断：

- `tokens/s` 更像系统总产能
- `TTFT` 更像首屏体感
- `TPOT` 更像生成过程流畅度
- `p95/p99` 决定“最差体验到底有多差”

## Benchmark 设计，至少要防住这些坑

### 基本固定项

- 固定硬件、驱动、CUDA / runtime / kernel 版本
- 固定模型版本、tokenizer 版本、sampling 参数
- 固定 warmup 策略、统计窗口和输入集

### 建议分桶

- 按输入长度分桶：短 / 中 / 长
- 区分 prefill 与 decode
- 区分单请求与并发请求

### 为什么要这样做

因为线上真实 workload 通常会混入：

- 排队
- 动态 batching
- 长短请求混跑
- KV cache 竞争
- allocator 碎片

只在单请求离线环境里测一把，往往只是“实验室成绩单”。

## 最小工程例子

假设你把模型从 FP16 切到 W8A16：

- 平均 `tokens/s` 提升 20%
- `p95 TTFT` 恶化 8%
- 某些任务 benchmark 下降 2 个点

这时“量化更快”这句话本身并不完整。

更准确的表述应该是：

- 吞吐提升了
- 首 token 体验未必更好
- 某些能力可能有回退

这正是 benchmark 必须多维汇报的原因。

## Troubleshooting：为什么离线 benchmark 好，线上体验却不一定好

| 现象 | 第一怀疑点 | 如何验证 |
|---|---|---|
| 单机吞吐很好，线上 p99 很差 | 并发、排队、动态 batching 没被覆盖 | 增加并发测试与长度分桶 |
| 平均延迟变好，但投诉变多 | p95/p99 恶化 | 补看尾延迟和最差请求分布 |
| 同样模型两次 benchmark 对不上 | tokenizer / sampling / warmup 不一致 | 固定配置并重跑 |
| 新优化离线收益明显，上线收益弱 | cache、allocator、KV 竞争影响更大 | 联合系统指标和 profile 复核 |

### 一个排障顺序

1. 先确认 benchmark 的环境和配置是否真的固定。
2. 再确认有没有按长度、并发、阶段分桶看结果。
3. 然后把离线收益和线上 workload 一一对齐。
4. 最后再讨论“这个优化值不值得上线”，别先被单一数字骗走。

## 推理优化工程师视角

这页最关键的作用，是防止你“只优化自己最想看的指标”：

- kernel 工程师容易只看单算子耗时
- serving 工程师容易只看 tokens/s
- 模型工程师容易只看任务集得分

但真正能指导上线决策的，通常是一组联合指标：

- 质量是否守住底线
- TTFT / TPOT / p99 是否可接受
- 显存、并发能力、GPU 利用率是否匹配目标成本

所以做任何优化前，最好先问清楚：

1. 是为了提升单机吞吐？
2. 是为了压低首 token 延迟？
3. 是为了降低显存，让更大 batch 或更长上下文可运行？

没有这个目标约束，benchmark 很容易变成一张漂亮但没决策价值的数字表。

## 面试高频问法

### 初级

1. throughput、TTFT、TPOT 分别描述什么？
2. 为什么 benchmark 需要固定硬件、软件版本和 warmup 策略？

### 中级

1. 为什么只看平均延迟远远不够？
2. 为什么离线 benchmark 好，不代表线上体验一定好？

### 高级

1. 如果一个优化让 tokens/s 提升，但 p99 恶化，你如何判断是否值得上线？
2. 如果两个版本质量接近，但一个显存更低、尾延迟更稳，你会怎么做取舍？

## 易错点

- 只做单条请求 benchmark，和真实线上差很大
- 忽略 warmup、cache、编译缓存等状态因素
- 只追单一 benchmark 分数，忽略真实 workload 和行为约束
- 把训练指标、能力指标、系统指标揉成一个“总分”

## 排查 checklist

- [ ] benchmark 脚本是否固定了版本、参数与输入集？
- [ ] 是否按长度分桶并报告 p95 / p99？
- [ ] 是否分别看了 prefill 与 decode？
- [ ] 是否记录了 GPU / CPU / 显存曲线？
- [ ] 是否能把离线结果映射回线上 workload？
