# 训练资源核算：参数、激活、梯度、优化器状态（选修）

## 要点

- 训练显存通常不止参数，还包括：**激活、梯度、优化器状态、临时工作区、通信 buffer**。
- 对 decoder-only Transformer，常用粗估：总训练 FLOPs 约为 $6NT$（预算级判断）。
- 真正有用的资源核算，不是背公式，而是能快速判断：**是该减 batch、减 context，还是必须上 checkpointing / sharding / 多卡并行**。
- 训练资源核算最容易犯的错，是把“模型大小”误当成“训练成本”，忽略了激活、状态和并行策略带来的二次放大。

## 通用知识

### 它是什么

训练资源核算是在回答三本账：

1. 显存账：单卡能不能装下
2. 吞吐账：每秒能跑多少 token / step
3. 总预算账：整个训练 run 要多少 FLOPs、多少卡时

### 它解决什么问题

它帮助你在训练前或训练中判断：

- 为什么会 OOM
- 为什么吞吐不够
- 为什么某种优化器或上下文长度设置突然不可行

### 为什么在 AI 系统里重要

因为很多架构选择并不只是“模型效果如何”，还隐含着：

- 激活会不会爆
- 优化器状态会不会装不下
- 长上下文下 attention 中间量会不会失控

### 它的收益与代价

收益：

- 让你能快速做训练配置决策
- 帮助判断该优先改哪里

代价：

- 公式通常是粗估，不是最终账单
- 不同实现、精度和并行策略会让真实开销偏离纸面值

### 三本账为什么要分开看

这是最值得反复强调的一点：

1. 显存账回答“能不能装下”
2. 吞吐账回答“能不能在可接受时间内训练”
3. 总预算账回答“这个项目从资源和时间上值不值得做”

很多训练决策之所以混乱，就是因为这三件事被混成了一个“感觉上能跑”。

例如：

- 一个配置可能显存上装得下，但吞吐过低
- 另一个配置吞吐还可以，但总卡时预算不可接受
- 还有的配置理论 FLOPs 不夸张，却会因为长上下文激活直接 OOM

## 最小显存心智模型

$$
M_{train} \approx M_{weights} + M_{grads} + M_{opt} + M_{acts} + M_{workspace}
$$

把它翻成口语化判断就是：

- 权重只是门票，激活和优化器状态才经常是大头

## 资源账的主要组成

### 1. 权重

- 模型参数本身占据的内存
- 混合精度训练时，可能还会保留更高精度 master weights

### 2. 梯度

- 通常与参数同阶
- 但保存精度和生命周期会影响真实占用

### 3. 优化器状态

- 以 AdamW 为例，常见至少还有一阶矩和二阶矩
- 所以“优化器比模型还占地方”不是段子，而是常态

### 4. 激活

- 与 batch、sequence length、hidden size、layers 强相关
- 长上下文下往往是最容易突然变大的部分

### 5. 临时工作区与通信 buffer

- kernel workspace
- collectives buffer
- allocator 预留和碎片

这些项单个看起来不一定最大，但临门一脚时很容易把训练从“能跑”推到 OOM

## 最小例子

假设模型参数量为 $P$：

- 权重本身占一份
- 梯度通常又是一份同阶量级
- AdamW 还会额外带两份动量/方差状态

所以即便先不算激活，训练显存也远不只是“模型参数大小”这么简单。

这就是为什么：

- 一个看起来“只有几 GB 权重”的模型
- 训练时却可能需要远大于这个数量级的显存

再给一个更具体的数量级直觉：

- 假设参数权重是 7B 量级
- 只看 BF16 权重本身，好像已经很大
- 但一旦加上梯度、AdamW 状态、激活和 workspace，训练显存很快就不再是“权重大小乘个 2”这么简单

这就是为什么很多新手会说：

- “模型文件明明能装进几张卡，为什么训练还是直接炸了？”

## 工程例子

一个常见场景：

- 模型参数量变化不大
- 只是把 context 从 4k 提到 16k
- 结果训练吞吐大幅下降，甚至直接 OOM

原因通常不是权重变大，而是：

- 激活显存迅速上升
- attention 中间量更贵
- micro-batch 被迫减小，吞吐进一步下降

这也是为什么长上下文训练往往不是“把窗口调长”那么简单。

再补一个常见判断：

- 如果你把 batch 从 8 改成 4 就能跑
- 但把 context 从 16k 改回 8k 之后吞吐和稳定性恢复得更明显

那问题通常更像：

- 激活和 attention 中间量主导

而不是：

- 权重本身装不下

## Activation Checkpointing 在资源账里做了什么

它的本质不是“免费省显存”，而是：

- 少存一部分激活
- 在反向传播时重算部分前向

也就是典型的交换：

- 显存下降
- 计算量上升
- 吞吐通常下降一些

因此是否开启 checkpointing，不能只看 OOM 是否解决，还要看：

- 训练时间是否还能接受
- 是否把瓶颈从显存挪到了总时长

## 混合精度与 sharding 在资源账里做了什么

### 混合精度

常见收益：

- 权重、激活、通信占用下降
- 某些硬件上吞吐提升

但要注意：

- 优化器状态未必按同样比例下降
- 数值稳定性和 loss scaling 仍然要管

### ZeRO / FSDP / Sharding

它们解决的是：

- 把参数、梯度、优化器状态分散到多卡上，降低单卡常驻占用

但代价是：

- gather / reduce-scatter / all-gather 等通信更多
- 吞吐不一定自然变好

所以它们主要回答的是“装不装得下”，不是“默认更快”。

## 推理优化工程师视角

即便你主攻推理，也建议至少会看懂训练资源账，因为很多推理相关设计来自训练约束：

- 为什么某些模型 hidden size / 层数 / context 这么选
- 为什么某些长上下文模型训练成本异常高
- 为什么某些结构要依赖 checkpointing、FSDP 或更激进的并行切分

如果只看推理侧的模型权重大小，而看不见训练侧的资源压力，很多架构判断就会显得“只看结果，不懂原因”。

## 常见面试问题

### 初级

1. 训练显存通常由哪些部分组成？
2. 为什么 AdamW 往往比只看权重大小更占显存？

### 中级

1. 为什么 context length 增大时，训练资源压力会迅速上升？
2. activation checkpointing 在资源账上做了什么交换？

### 高级

1. 如果训练 OOM，你会优先减 batch、减 context，还是上 checkpointing？为什么？
2. 为什么 $6NT$ 这种公式适合预算级估算，但不适合直接拿来当精确账单？
3. 为什么有时模型参数变化不大，但训练总成本会突然陡增？

## 常见误区

- 只按模型权重大小估算训练显存
- 忽略优化器状态开销（AdamW 常显著增大显存）
- 把 checkpointing 当成纯收益，不算重算代价
- 把吞吐问题和显存问题混成一件事，导致优化方向跑偏

## 排查 checklist

- [ ] 能否分别给出 weights / grads / optimizer / activations 的数量级？
- [ ] OOM 更像参数装不下，还是激活太大？
- [ ] 减 batch、减 context、开 checkpointing 哪个最直接？
- [ ] 当前问题是显存账、吞吐账，还是总预算账先出问题？
