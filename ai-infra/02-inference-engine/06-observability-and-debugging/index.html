    可观测性与调试（profiling、tracing、metrics）¶

    要点¶

    • 没有可观测性就很难做性能/稳定性闭环
    • 线上最有用的不是“平均值”，而是分布与分桶（长度、并发、模型版本）
    • 真正成熟的推理优化，不是“猜哪里慢”，而是能把问题定位到：哪一段链路、哪类请求、哪一版变更、哪一个热点算子。

    通用知识¶

    它是什么¶

    可观测性回答的是：

    • 系统当前发生了什么
    • 问题发生在什么阶段
    • 变更后到底哪里变好了、哪里变差了

    调试则回答：

    • 如何把“系统感觉不对”一步步缩小到具体模块、具体请求、具体 kernel 或具体变更

    它解决什么问题¶

    它主要解决三类问题：

    1. 性能问题：TTFT、TPOT、吞吐、p99 为什么异常
    2. 稳定性问题：超时、OOM、偶发抖动、慢请求集中出现在哪里
    3. 回归问题：某次版本上线后，到底是哪一环退化了

    为什么在 AI 系统里重要¶

    因为 LLM serving 的问题几乎都不是“总时间慢”这么简单，而是会混在一起：

    • 排队
    • tokenize
    • prefill
    • decode
    • 后处理
    • cache / allocator 抖动
    • 编译或 cold start

    没有可观测性，优化很容易变成：

    • 看到 attention 慢就换 kernel
    • 看到 GPU 利用率低就盲目增 batch
    • 最后既没治根因，还顺手制造了新问题

    它的收益与代价¶

    收益：

    • 让定位更快
    • 让回归更可追踪
    • 让优化动作能被证实或证伪

    代价：

    • 指标体系、trace 体系和 profiling 方法都需要持续维护
    • 指标埋点过粗会没用，过细又会增加系统复杂度和数据成本

    Metrics（建议你后续按项目补）¶

    • 请求：QPS、排队时间、超时率、错误码
    • 延迟：TTFT、TPOT、p50/p95/p99
    • 资源：GPU utilization、SM occupancy（如果可得）、显存占用、CPU 利用率
    • 质量：回归样例通过率、关键业务指标

    再补一个很关键的原则：

    • 所有核心指标尽量都要能按 输入长度 / 输出长度 / 并发水平 / 模型版本 / 采样参数 分桶

    否则平均值会很容易把问题洗平。

    一个够用的观测框架¶

    如果你只想记一个最小框架，可以记成四层：

    1. 请求层：QPS、排队、超时、错误率
    2. 延迟层：TTFT、TPOT、p95/p99、按长度分桶
    3. 资源层：GPU/CPU/显存/带宽
    4. 变更层：模型版本、runtime 版本、量化配置、采样参数、流量形态

    只有把这四层串起来，才有机会回答：

    • “慢”到底是 workload 变了，还是实现变了

    Tracing¶

    • 关键 span：tokenize、prefill、decode 循环、postprocess
    • 关联：请求 ID、模型版本、采样参数、输入长度

    更工程化一点，trace 最重要的价值是：

    • 把单个请求的完整生命周期串起来

    这样你能区分：

    • 是请求在队列里等太久
    • 还是 prefill 真算太久
    • 还是 decode 某一步被 cache/allocator 拖慢

    Profiling¶

    • CPU：热点函数、线程争用
    • GPU：kernel 时间分布、kernel 次数、带宽/占用

    一个常见误区是把 profiling 和 metrics 混成一件事。

    更准确地说：

    • metrics 用来看整体趋势和线上分布
    • trace 用来看一次请求的完整链路
    • profiler 用来看函数、算子、kernel 到底慢在哪

    三者是互补关系，不是替代关系。

    最小例子¶

    假设线上用户反馈：

    • 首 token 很慢
    • 但开始出 token 后速度还行

    如果你只有平均 latency，很难下手。

    但如果你同时有：

    • TTFT / TPOT 分离指标
    • prefill / decode trace
    • 输入长度分桶

    你就能很快判断：

    • 问题更像在 queue 或 prefill
    • 而不是 decode attention 本身

    这就是为什么指标拆解比“总耗时”重要得多。

    工程例子¶

    一个很常见的线上场景：

    • 平均 tokens/s 没明显下降
    • 但 p99 延迟突然变差

    这时最危险的误判是：

    • “既然平均吞吐没事，系统应该没问题”

    实际上更可能是：

    • 长短请求混跑变严重
    • allocator 抖动或 cache 回收导致尾部请求受伤
    • 某次变更只影响了少数长度桶

    如果你的指标没有按长度和并发分桶，这类问题几乎一定会被平均值掩盖。

    再给一个 profiler 视角的例子：

    • GPU utilization 看着不低
    • 但 kernel 次数异常多、单次很碎

    这通常提示：

    • launch overhead 或融合不足在伤你

    而不是简单的“GPU 很忙所以没问题”。

    变更回归该怎么做¶

    一个够用的回归闭环通常包括：

    1. 固定一组代表性 workload
    2. 固定模型版本、runtime 版本、量化配置和采样参数
    3. 保留关键指标基线：TTFT、TPOT、吞吐、p95/p99、显存
    4. 一旦退化，先看哪一层先变差：请求层、延迟层、资源层还是 kernel 层

    这样你才不会在一次版本回归里陷入“感觉哪都不太对”的泥潭。

    推理优化工程师视角¶

    如果说优化 Playbook 回答的是“慢了该怎么做”，那可观测性回答的就是“你凭什么确信自己找对了地方”。

    对推理优化工程师来说，这篇最重要的 4 个本能是：

    1. 指标必须拆阶段，不能只看总耗时
    2. 指标必须分桶，不能只看均值
    3. trace 必须能串单请求，不能只看聚合报表
    4. profiler 必须能落到热点函数和 kernel，不能停留在“GPU 利用率挺高”

    常见面试问题¶

    初级¶

    1. 为什么推理系统不能只看平均延迟？
    2. metrics、trace、profiler 分别适合解决什么问题？

    中级¶

    1. 如果 TTFT 高、TPOT 正常，你会优先看哪些指标和 span？
    2. 为什么输入长度和并发水平分桶很关键？

    高级¶

    1. 如果平均吞吐正常但 p99 很差，你会如何构建定位路径？
    2. 如果 GPU utilization 不低，但整体性能仍不好，你会优先怀疑什么？

    易错点¶

    • 指标没有按输入长度/输出长度分桶，导致看不出问题
    • 只看 GPU 利用率，不看 kernel 数/带宽/同步点
    • 只看聚合指标，不建立请求级 trace
    • 版本回归时没有同时固定 workload、模型版本和采样配置

    排查 checklist¶

    • [ ] 能否用一次 trace 把单请求的全链路还原？
    • [ ] 是否能定位到具体 kernel/算子热点？
    • [ ] 是否能把性能回归关联到一次版本变更？
    • [ ] 指标是否按输入长度、输出长度、并发和模型版本分桶？
