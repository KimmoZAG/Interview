    Paged KV 与 Allocator：为什么 KV cache 不是“开个数组”这么简单¶

    要点¶

    • 从 CS336 Lecture 10 的工程落点看，KV cache 不只是一个模型概念，而是推理系统最关键的状态对象之一。
    • 一旦进入真实 serving，问题立刻从“KV 要不要存”变成“怎么分配、怎么复用、怎么回收、怎么避免碎片”。
    • paged KV 的核心价值不是概念新奇，而是让长短不一、动态到达的请求可以更稳定地管理 KV 状态。
    • allocator 设计会直接影响：显存碎片、扩容成本、尾延迟、请求驱逐和系统稳定性。

    通用知识¶

    它是什么¶

    在自回归 LLM 推理里，KV cache 是每层 attention 的状态缓存；paged KV 则是管理这份缓存的一种更工程化的方法。

    相比“每个请求给一整段连续空间”的朴素做法，paged KV 的思路是：

    • 把 KV 拆成固定大小 page / block
    • 请求只维护逻辑位置到物理块的映射

    这样问题就从“缓存存哪里”升级成了：

    • 块怎么分配
    • 块怎么回收
    • 映射怎么维护
    • allocator 怎样避免抖动

    它解决什么问题¶

    它主要解决：

    • 长短请求长度差异很大时，连续布局扩容和碎片化严重
    • continuous batching 下，请求生命周期动态变化
    • 长上下文和高并发会把 KV cache 变成显存主力

    为什么在 AI 系统里重要¶

    因为 decode 阶段的很多瓶颈都不是“attention 公式太慢”，而是：

    • 历史 KV 太大
    • KV 块管理不稳定
    • allocator 频繁分配释放
    • block 粒度和请求分布不匹配

    它的收益与代价¶

    收益：

    • 内存增长与回收更灵活
    • 更适合动态请求生命周期
    • 更容易降低大块连续显存需求

    代价：

    • 映射与索引更复杂
    • kernel 读取路径更复杂
    • allocator 设计会直接进入性能关键路径

    1. 为什么连续 KV 布局会出问题¶

    最直观的做法是：

    • 为每个请求分配一段连续显存
    • 把每一层的 K/V 依次写进去

    这在静态 batch 或短请求场景下很好理解，但真实 serving 很快会出现问题：

    • 不同请求长度差异很大
    • 请求到达和结束时间不一致
    • 有的请求生成很短，有的请求生成很长

    于是连续布局容易带来：

    • 扩容拷贝
    • 显存空洞
    • 碎片累积

    这也是为什么很多“理论上显存够”的系统，线上仍然会出现：

    • 偶发 OOM
    • p99 抖动
    • 显存利用率看起来不低，但吞吐不稳定

    2. Paged KV 在解决什么¶

    paged KV 的思路是：

    • 不要求每个请求独占一整段连续大块显存
    • 而是把 KV 按 page 或 block 切成固定粒度的小块
    • 请求只持有这些块的映射关系

    这样做的好处是：

    • 更容易增长和回收
    • 减少大块连续显存需求
    • 更适合 continuous batching 和动态请求生命周期

    一句更口语化的说法是：

    • 它把“每个请求要一整条连续长街”改成了“每个请求拿若干标准化车位”

    3. 为什么 allocator 会成为关键组件¶

    一旦用 page/block 管理 KV，系统就必须回答：

    • 新请求来了，从哪里拿 block
    • 请求结束后，哪些 block 可以回收
    • block 不够时，驱逐谁
    • 如何避免频繁分配释放造成抖动

    这就从“模型缓存”问题升级成了“内存分配器”问题。

    所以一旦进入 serving 真实环境，allocator 不再是幕后配角，而是会直接决定：

    • 系统抖不抖
    • tail latency 漂不漂
    • 长请求会不会把显存压穿

    4. 一个最小心智模型¶

    可以把 Paged KV 系统想成三层：

    层 1：逻辑 token 序列¶

    • 用户看到的是一串上下文 token

    层 2：KV pages / blocks¶

    • 系统把这些 token 对应的 KV 切成固定大小块

    层 3：映射和调度¶

    • 每个请求维护逻辑位置到物理 block 的映射
    • 调度器和 kernel 根据映射读取所需 KV

    5. Paged KV 的主要收益¶

    收益 1：更好的内存复用¶

    • 请求结束后可以只回收对应 block

    收益 2：更少的扩容拷贝¶

    • 序列变长时不必整体搬迁已有 KV

    收益 3：更适合动态 batching¶

    • 不同请求可以独立增长，不必预先按最大长度保留大块空间

    最小例子¶

    假设系统里同时有两类请求：

    • 短请求：输入 64 token，输出 32 token
    • 长请求：输入 8k token，输出 512 token

    如果采用连续大块分配：

    • 长请求更容易造成大块占用
    • 请求结束顺序不同会留下显存空洞
    • 新请求进来时可能需要重新找更大连续空间

    如果采用 paged KV：

    • 长请求只是多拿更多 block
    • 短请求结束后，对应 block 可以更细粒度回收
    • allocator 可以更灵活地复用空闲块

    这个例子最重要的直觉是：

    • paged KV 解决的不是“KV 需不需要缓存”，而是“KV 在动态服务系统里怎么活下去”

    6. 它的代价是什么¶

    Paged KV 不是免费午餐，它通常带来：

    • 地址映射开销
    • 更复杂的 kernel 读取逻辑
    • block 粒度带来的内部碎片
    • allocator 元数据与调度复杂度

    所以工程上不能只问“支不支持 paged KV”，还要问：

    • block 大小是否合理
    • 映射访问是否破坏了访存局部性
    • allocator 是否成为新的延迟来源

    block 大小为什么是关键参数¶

    block 太小：

    • 元数据和映射开销更大
    • 分配释放更频繁

    block 太大：

    • 内部碎片更严重
    • 短请求可能浪费很多空间

    所以 block 大小本质上是在平衡：

    • 映射复杂度
    • 内部碎片
    • 访存局部性
    • 请求长度分布

    7. 什么时候 allocator 会伤你最深¶

    最容易出问题的场景包括：

    • 长短请求混跑
    • 高频突发流量
    • 多模型或多租户共享 GPU
    • 长上下文请求占满大量 KV 页

    在这些场景下，allocator 设计不好会直接表现为：

    • p99 抖动
    • 偶发 OOM
    • throughput 不稳定

    工程例子¶

    一个典型现象：

    • 平均吞吐看起来正常
    • 但线上偶尔出现长尾请求明显慢很多

    这时如果你只看 attention kernel，很容易看不到问题。

    更真实的根因可能是：

    • allocator 在高并发下频繁分配释放 block
    • 长请求占住大量 page，驱逐与回收策略变得激进
    • block 粒度与真实流量长度分布不匹配

    这种问题的典型特点是：

    • 平均值未必难看
    • p99 和显存行为会先开始发脾气

    8. 应该观测哪些指标¶

    除了常规的 TTFT、TPOT、吞吐，还应特别盯：

    • 活跃 KV 页数
    • block 利用率
    • 分配/释放频率
    • 请求驱逐率
    • 显存碎片与保留内存

    如果不观测这些，你就只能看到“系统偶尔抖”，但无法定位是调度问题还是内存管理问题。

    推理优化工程师视角¶

    如果你做的是推理优化，这篇最关键的价值是提醒你：

    1. KV cache 不只是模型概念，更是系统状态对象
    2. decode 变慢时，不要只盯 attention kernel，也要怀疑 KV 布局和 allocator
    3. p99、OOM 和吞吐抖动很多时候是内存管理问题，不是单纯算力问题
    4. 任何 paged KV 方案都要结合请求长度分布来看，不存在绝对最优参数

    9. 和长上下文有什么关系¶

    上下文越长：

    • KV cache 越大
    • allocator 压力越高
    • page/block 数量越多

    因此长上下文推理几乎一定会把 KV 管理和内存分配推到系统核心位置。

    易错点¶

    • 把 KV cache 只当成 attention 的附属缓存
    • 只比较总显存占用，不看碎片和分配行为
    • 只知道 paged KV 更灵活，却说不清它增加了哪些映射和调度成本
    • 不做分桶分析，就把所有线上抖动归因给模型本身
    • 只知道 page/block 化能省事，却不去量化 block 大小和长度分布是否匹配

    排查 checklist¶

    • [ ] 当前 KV 布局是连续、分页，还是混合方案？
    • [ ] block 大小是否和请求长度分布匹配？
    • [ ] 是否记录了 allocator 相关指标，而不只是总显存？
    • [ ] p99 抖动时，究竟是 decode、分配器，还是驱逐策略在出问题？
