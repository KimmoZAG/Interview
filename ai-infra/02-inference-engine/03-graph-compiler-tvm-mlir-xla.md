# 图编译：TVM / MLIR / XLA（概念对齐）

## 要点

- 图编译的目标：把“高层算子图”变成“更少、更快的 kernel/代码”，并适配目标硬件
- 你需要把三个概念分清：**IR 表达 → 优化 pass → 代码生成/调度**
- 图编译真正有价值的地方，不是“名词很多”，而是它提供了一条从模型图到硬件执行之间的可控优化链路。

## 通用知识

### 它是什么

图编译可以理解为：

- 先把高层模型表示成中间表示（IR）
- 再对 IR 做一系列 pass
- 最后 lower 到更接近硬件的代码或执行计划

### 它解决什么问题

它要解决的是：

- 高层模型图太粗，不够接近硬件
- 手写每个 kernel 不现实
- 需要针对不同 shape、layout、硬件做系统性优化

### 为什么在 AI 系统里重要

因为很多性能收益并不来自改模型结构，而来自：

- 常量折叠
- op 融合
- layout 重排
- lowering 和 codegen 更适合目标硬件

### 它的收益与代价

收益：

- 降低 kernel 数量
- 提高数据复用
- 更容易适配新硬件和新后端

代价：

- 编译时间更长
- 动态 shape 处理更复杂
- 调试和回归定位难度上升

## 共同概念（不绑定实现）

- IR：中间表示（算子图/循环/张量表达等）
- Pass：融合、常量折叠、dead code elimination、layout 变换
- Lowering：从高层算子逐步降到更接近硬件的表示
- Codegen：生成 CUDA/HIP/CPU 向量化代码等

## 最小例子

考虑一个简单子图：

- `matmul -> bias add -> gelu`

如果没有编译优化：

- 可能变成多个独立 kernel

如果图编译器做了融合：

- 可能把 bias add 和 gelu 融进更少的执行单元
- 中间张量写回减少
- kernel launch 次数减少

这就是为什么图编译常常和“更少的中间结果”和“更少的 launch”绑在一起。

## 工程落地关注点

- 动态 shape：需要 shape-specialization 或多版本 cache
- 自动调优：代价与收益（编译时间 vs 运行收益）
- 可调试性：IR dump、pass 可视化、回归定位

## 工程例子

一个典型线上问题是：

- 某个 shape 跑得很快
- 一旦 shape 稍微变化，性能突然回退

这通常意味着：

- 当前最优路径依赖 shape specialization
- 编译缓存只覆盖了部分 shape
- 或者某些 pass 在动态 shape 下失效

所以图编译问题经常不是“编译器不够聪明”，而是“真实流量 shape 分布比 benchmark 复杂得多”。

## 推理优化工程师视角

推理优化工程师读图编译，不是为了背术语，而是为了学会把“性能收益从哪里来”拆开看：

- 是因为 kernel 数变少了
- 是因为 layout 更顺了
- 是因为某些 shape 被专门优化了
- 还是只是把成本转移到了编译时和冷启动阶段

这很重要，因为图编译经常会制造一种错觉：离线最优路径很漂亮，线上真实收益却不稳定。真正要关注的是“收益是否能被真实流量重复拿到”。

一个很实用的工程习惯是同时看三张账：

1. 编译时间账
2. 缓存命中账
3. 运行时 kernel / latency 账

只看其中一张，往往会得到很乐观但不耐用的结论。

## 常见面试问题

### 初级

1. IR、pass、lowering、codegen 分别是什么？
2. 为什么图编译可以减少 kernel 数？

### 中级

1. 为什么动态 shape 会让图编译和缓存策略变复杂？
2. 图级融合和 kernel 级融合有什么联系和区别？

### 高级

1. 如果编译后某些 shape 很快、另一些很慢，你会从哪里开始定位？
2. 为什么图编译收益不能只看单次运行时间，还要看编译成本和冷启动？

## 易错点

- 编译缓存策略不当导致线上抖动（冷启动慢）
- 过度 specialization 导致 cache 爆炸
- 只看最终吞吐，不看是否为了性能付出了不可接受的编译时延
- 只看高层 IR，不验证底层 kernel 是否真的变少了

## 排查 checklist

- [ ] 编译耗时与运行收益是否量化？
- [ ] 是否能稳定复现某个 shape 的最优路径？
- [ ] 是否有 IR/pass 级别的回归对比手段？
