# Post-training 与 Alignment：SFT、偏好优化、RL 的最小框架

## 要点

- 预训练后的 base model 还不等于可用 assistant。
- 常见后训练链路：SFT → 偏好优化/RL → 行为与系统评测闭环。
- alignment 既是行为问题，也是系统问题（数据、奖励、推理配置共同决定）。

## 通用知识

### 它是什么

post-training 指的是在预训练之后，为了把 base model 变成更可用、更符合目标行为的 assistant，而做的一系列训练与优化步骤。

最常见的框架包括：

- SFT：先把模型拉到“能对话、能遵循指令”的区域
- 偏好优化：让模型更倾向产生被人偏好的回答
- RL 类方法：在奖励信号下继续优化策略

### 它解决什么问题

预训练模型通常具备通用建模能力，但不天然等于：

- 回答方式稳定
- 指令遵循良好
- 安全性与行为边界可控

post-training 解决的正是“会预测 token”到“更像可用 assistant”之间的差距。

### 为什么在工程里重要

因为很多用户真正感受到的模型差异，不只来自预训练规模，也来自后训练阶段怎么塑造行为。

同时它也会带来典型 trade-off：

- 更对齐，但可能伤基础能力
- 更安全，但可能更保守
- 更会遵循格式，但可能成本更高或输出更长

## 最小框架

1. SFT：把模型拉到可用 assistant 区域
2. Preference Optimization：学习“哪个回答更好”
3. RL-based Post-training：在 reward 信号下优化策略分布
4. 多维评测闭环：能力、行为、系统成本一起看

## 最小例子

一个典型演化过程是：

- 预训练模型能续写，但回答风格不稳定
- 加了 SFT 后，能更像助手一样答题
- 加了偏好优化后，输出更符合人类偏好
- 再继续强化某些行为目标时，可能出现“更安全但更保守”的现象

这说明 alignment 从来不是单向变好，而是在不同目标之间重新分配模型行为。

## 工程例子

一个常见项目现象是：

- 某次后训练后，人工主观评价变好了
- 但某些基础 benchmark 下降，或者生成长度和成本上升

这时如果只从“偏好分数更高”下结论，往往会误判。

更稳妥的做法通常是：

- 同时看基础能力、行为指标、成本和系统延迟
- 检查 reward 或偏好数据是否把模型推向了过度保守/冗长的回答
- 固定推理配置，避免把 sampling 差异误当作 alignment 收益

换句话说，alignment 结果必须放在系统上下文里看，不能只看一张偏好表。

## 常见面试问题

### 初级

1. 为什么预训练后的 base model 还不等于可用 assistant？
2. SFT、偏好优化、RL 类后训练各自大致在解决什么问题？

### 中级

1. 为什么 alignment 常常伴随能力、行为和成本之间的 trade-off？
2. 为什么评估后训练效果时不能只看单一偏好指标？

### 高级

1. 如果某次对齐训练后用户偏好提升，但基础能力和延迟恶化，你会如何做决策？
2. 你会如何区分“模型真的更 aligned”与“只是推理配置变了所以看起来更稳”？

## 为什么放在附录

这篇和大模型架构、评测体系、产品指标关系很深，但不属于推理优化主线本身。

它放在附录更合适，因为：

- 它是理解 assistant 行为形成的关键背景知识
- 它经常作为面试里的延伸追问出现
- 它帮助你把训练、评测、产品体验三者联系起来

## 常见风险

- 对齐增强但基础能力回退
- reward 学歪（reward hacking）
- 评测过于单一导致误判
- 推理配置变化后效果失稳

## 排查 checklist

- [ ] 当前阶段是 SFT、偏好优化还是 RL？
- [ ] 监督信号/reward 来源是否可靠？
- [ ] 是否同时观察能力、行为、成本？
- [ ] 推理配置变化后效果是否稳定？
