# 预训练数据工程：来源、过滤、去重、质量评估

## 要点

- 数据工程是模型能力边界与训练成本的共同决定因素。
- 最小流水线：来源 → 解析 → 过滤 → 去重 → 混配 → 质量评估。
- 目标不是“尽可能多抓数据”，而是提高有效 token 密度。

## 通用知识

### 它是什么

预训练数据工程关注的不是“抓了多少网页”，而是怎样把原始、多噪声、分布混乱的数据，整理成一份能稳定支撑训练的语料。

它通常包括：

- 来源选择
- 抽取解析
- 清洗过滤
- 去重
- 质量评估
- 配方管理

### 它解决什么问题

它要解决的是：

- 低质量 token 太多，白白消耗训练预算
- 重复数据太多，导致模型“看起来学得很快”，但泛化不一定更好
- 目标能力缺失，导致模型偏科

### 为什么在工程里重要

因为训练成本很高，而数据质量问题常常是“最贵、最隐蔽”的问题之一。

同样的训练预算，如果有效 token 密度更高，往往能换来：

- 更好的收敛质量
- 更少的无效计算
- 更稳定的能力分布

## 最小流水线

1. 数据来源管理（网页/代码/文档/对话等）
2. 解析抽取（去 boilerplate、语种识别、清洗）
3. 过滤（质量、长度、合规）
4. 去重（精确 + 模糊）
5. 混配与切分
6. 小实验验证（loss/benchmark）

## 最小例子

假设你有两份网页语料：

- 语料 A：总量大，但模板页、导航栏、重复内容很多
- 语料 B：总量小，但正文密度高、结构更干净

如果只看 token 总数，A 很诱人；但从训练价值看，B 可能更划算。

这正说明一个关键点：

- 数据工程优化的目标不是“让 token 变多”
- 而是“让每个 token 更值得喂给模型”

## 工程例子

一个常见真实场景是：

- 团队扩充了大量新数据源
- 训练 loss 看起来下降更快
- 但后续 benchmark 提升不明显，甚至部分能力回退

常见根因包括：

- 新数据源重复率高
- boilerplate 和低信息密度内容比例过高
- 清洗规则过松，噪声被大量引入

这类问题如果只看数据量，很难及时发现；必须配合抽样检查、去重统计和小规模训练回路一起看。

## 常见面试问题

### 初级

1. 预训练数据工程的最小流水线通常包含哪些步骤？
2. 为什么“更多数据”不一定等于“更好数据”？

### 中级

1. 精确去重和模糊去重分别解决什么问题？
2. 为什么数据工程改动不能只靠规则上线，还要做抽样和小实验验证？

### 高级

1. 如果某次数据扩充后 loss 更好了，但下游能力没提升，你会优先排查哪些环节？
2. 你会如何判断一个新数据源是否值得进入主训练配方？

## 为什么放在附录

这篇很重要，但它不属于推理优化主线。

它更像是：

- 解释模型能力为什么会长成现在这样
- 帮你在面试中把“数据质量”和“训练收益”联系起来
- 帮你理解训练阶段的问题如何传导到后续系统和产品表现

## 易错点

- 只追数据量，不追质量密度
- 只做精确去重，不做模糊去重
- 规则很多但没有抽样与回归验证

## 排查 checklist

- [ ] 各数据源分布是否符合目标任务？
- [ ] 过滤与去重比例是否被持续记录？
- [ ] 是否做过抽样人工检查？
- [ ] 是否有小规模训练回路验证改动收益？
