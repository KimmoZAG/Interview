# 硬件互联：PCIe / NVLink / NVSwitch / InfiniBand / RDMA

## 要点

- 通信优化的第一原则不是“把代码写得更花”，而是先确认数据究竟走哪条链路。
- 同样是 GPU 间通信，**同机箱 NVLink/NVSwitch** 和 **跨机 InfiniBand/RDMA** 的性能与抖动特征完全不同。
- 拓扑决定的不只是峰值带宽，还决定了是否会出现热点链路、绕路、oversubscription 和尾延迟放大。

## 通用知识

### 它是什么

互联硬件回答的是：设备之间通过什么物理链路交换数据。

在大模型系统里，常见链路包括：

- PCIe：通用外设总线，GPU/CPU/NIC 都可能挂在上面
- NVLink：GPU 到 GPU 的高速点对点链路
- NVSwitch：让多张 GPU 通过交换芯片形成更高带宽互联
- InfiniBand：常见于跨机器高性能集群网络
- RDMA：让一台机器直接访问另一台机器内存，绕过更多 CPU 参与

### 它解决什么问题

它决定了：

- 一次 collective 能跑到什么上限
- 哪些并行策略能接受，哪些会把通信放大到不可用
- 同机多卡和跨机多卡的扩展曲线为什么经常长得不一样

### 为什么在 AI 系统里重要

因为很多训练/推理方案理论上只差一个 collective，但工程上差的是 collective 落在了哪种链路上：

- TP 常常要求更频繁、更低延迟的 GPU-GPU 通信
- FSDP/ZeRO 常常把更多通信放在参数 gather / shard 上
- MoE 的 all-to-all 对拓扑尤其敏感

### 它的收益与代价

- 更强的互联：更高带宽、更低延迟、更多可行并行策略
- 更复杂的拓扑：排障难度更高，热点链路和非对称路径更难肉眼发现

## 最小例子

假设有 8 张 GPU：

- 如果它们通过 NVSwitch 高速互联，做 TP 时每层的 all-gather / reduce-scatter 代价可能还能接受
- 如果它们只是通过 PCIe 挂在不同 root complex 下，同样的 TP 可能直接把每层通信变成瓶颈

所以“8 卡 TP 能不能跑得好”这个问题，答案不只取决于模型大小，也取决于 8 卡之间到底怎么连。

## 工程例子

一个常见现象是：

- 单机 8 卡吞吐不错
- 扩到 2 机 16 卡后扩展效率突然明显掉下去

这时常见根因不是代码突然变差，而是：

- 原本大量通信都走机内 NVLink/NVSwitch
- 扩机后，一部分 collective 被迫跨机走 InfiniBand
- 若 bucket 太小、同步太频繁，延迟开销就会被放大

## 推理优化工程师视角

推理优化里，很多“同样模型、同样卡数，表现却差很多”的现象，本质上都要回到这一章。

- 单机多卡和多机多卡不是同一种问题
- 拓扑不只是背景信息，而是决定方案能否成立的前提条件
- 你选择 TP、EP、paged KV 跨卡共享还是分片部署，都会受链路约束

所以工程上一个很重要的习惯是：在讨论优化方案前，先把设备连接关系画出来，至少明确：

1. 热点通信走机内还是跨机
2. 哪些 rank 之间的路径最贵
3. 是否存在 NIC / NUMA / root complex 绑定不合理

很多性能问题并不是“实现不够高级”，而是方案默认了某种拓扑条件，但机器根本不满足。

## 常见面试问题

### 初级

1. PCIe、NVLink、InfiniBand 分别大致解决什么问题？
2. 为什么多卡通信不能只看“GPU 数量”，还要看拓扑？

### 中级

1. 为什么 TP 通常比 DP 更依赖高速 GPU-GPU 互联？
2. 为什么同样 8 卡，不同机箱拓扑下性能差距可能很大？

### 高级

1. 如果 16 卡扩展效率从 80% 掉到 45%，你会怎么判断是链路问题还是软件实现问题？
2. 为什么 MoE 的 all-to-all 对拓扑往往比普通 all-reduce 更敏感？

## 易错点

- 把“有 8 张卡”误当成“有同样的通信能力”
- 只看理论峰值带宽，不看真实路径和拥塞
- 忽略 CPU root complex、NUMA 和 NIC 绑定关系

## 排查 checklist

- [ ] rank 到 GPU、GPU 到 NIC、NIC 到交换网络的映射是否清楚？
- [ ] 当前热点 collective 主要走机内还是跨机链路？
- [ ] 是否存在部分 rank 明显更慢，暗示拓扑不对称？
- [ ] 实测带宽与链路理论上限差多少？
