    并行策略与通信映射：DP / TP / PP / SP / FSDP / ZeRO¶

    核心定义（What & Why）¶

    一句话总结：这篇讲的不是并行策略定义本身，而是“选了哪种切分，就等于选了哪种通信形态”，它解决的是为什么相同模型在不同并行方案下会暴露完全不同的性能风险。

    关联知识网络¶

    • 前置：并行训练策略
    • 前置：Collectives
    • 平行：互联与拓扑
    • 课程桥接：CS336 / 07 并行训练（一）
    • 课程桥接：CS336 / 08 Collectives 与实现细节
    • 排障：推理优化 Playbook

    要点¶

    • 讨论并行策略时，最有价值的问题不是“定义是什么”，而是“它把通信放到了哪里”。
    • 同一种模型，在不同并行策略下，瓶颈 collective 的种类、频率和位置会完全不同。
    • 面试和工程里都很重要的一点是：能把 切什么、通信什么、什么时候同步 说成一张图。

    通用知识¶

    它是什么¶

    这篇不是重复定义 DP/TP/PP/FSDP，而是把它们翻译成通信语言：

    • DP：同步梯度
    • TP：同步层内局部结果
    • PP：同步 stage 边界激活
    • SP：同步序列维局部块
    • FSDP/ZeRO：同步参数、梯度或优化器状态分片

    它解决什么问题¶

    • DP：在模型装得下的前提下扩大 batch 和吞吐
    • TP：解决单层矩阵太大、单卡算不动/装不下
    • PP：解决模型过深
    • FSDP/ZeRO：解决状态内存太大

    为什么在 AI 系统里重要¶

    因为同一句“多卡训练太慢”，在不同策略下含义完全不同：

    • DP 慢，通常先看梯度同步
    • TP 慢，通常先看层内频繁 all-gather / reduce-scatter
    • PP 慢，通常先看 pipeline bubble 和 stage 负载不均
    • FSDP 慢，通常先看参数 gather / shard 时机与 overlap

    它的收益与代价¶

    • 更激进的切分通常能省显存
    • 但也通常意味着更高频、更复杂、更难 overlap 的通信

    最小例子¶

    一个很实用的对照表：

    策略 主要切分对象 常见通信对象 常见风险
    DP batch 梯度 all-reduce 成本高
    TP 层内矩阵 局部激活/输出 每层同步频繁
    PP 层段 stage 边界激活 bubble、负载不均
    FSDP/ZeRO 参数/梯度/优化器状态 参数分片、梯度分片 gather 时机不当
    SP 序列维 序列块相关激活 长上下文下同步复杂

    对比表：选并行策略时，通信风险首先落在哪¶

    策略 最重的通信通常发生在哪里 更依赖什么 最常先怀疑的风险
    DP step 末或 bucket 梯度同步 all-reduce 吞吐与 overlap global batch 太小、尾部 bucket 重
    TP 层内前后向关键路径 机内低延迟高速互联 每层同步太密、跨节点延迟高
    PP stage 边界 micro-batch 数和 stage 均衡 bubble、负载不均、边界传输抖动
    FSDP / ZeRO 参数 gather / 梯度 shard 时刻 all-gather / reduce-scatter 组织方式 gather 时机不当、无法 overlap
    SP 长序列局部块同步 序列切分策略 上下文越长越难控通信复杂度

    工程例子¶

    假设一个模型：

    • 8 卡 DP 可以跑，但 global batch 继续增大后 all-reduce 成本很高
    • 改成 TP 后显存问题缓解，但 step time 反而没下降

    可能原因不是 TP “无效”，而是：

    • 层内同步频率比 DP 高得多
    • 当前互联拓扑并不适合高频 TP 通信
    • 算子规模还没大到足以摊薄 TP 的同步开销

    💥 实战踩坑记录（Troubleshooting）¶

    现象：从 DP 切到 TP 后显存问题缓解了，但 step time 没降反升。

    • 误判：觉得“只要能把单层切开，性能自然就会更好”。
    • 根因：TP 省的是单层容量，但引入的是层内高频同步；如果拓扑不够强，或者单层规模还不足以摊薄通信，就会出现“容量变好，吞吐变差”。
    • 解决动作：
      • 先确认瓶颈是容量问题还是吞吐问题；
      • 再看 TP 的 collective 是否落在关键路径；
      • 最后判断是否应改为 TP 只留节点内、外层用 DP/FSDP/PP 组合。
    • 复盘：并行方案从来不是“更高级就更快”，而是“它的通信形态是否匹配当前硬件和 workload”。

    常见异常：FSDP 显存降下来了，但训练吞吐与稳定性都变差。

    • 这往往说明问题已经切换到 gather/reshard 时机、bucket、prefetch 和 overlap 组织上，而不是单纯“显存终于够了”就结束了。

    推理优化工程师视角¶

    这篇其实是通信板块里最接近“性能建模”的一篇。对推理优化工程师而言，它的价值在于把并行方案从名词表，变成成本映射表：

    • 选择一种切分，本质上是在选择一种通信形态
    • 通信形态一旦确定，风险点基本也跟着确定
    • 后面的 profiling、容量评估、上线决策，都应该围绕这张映射表展开

    因此在实际工作里，一个很强的习惯是：遇到新方案时，不先问“它有没有人用过”，而先问：

    1. 它让哪些张量跨卡流动？
    2. 这些流动发生在层内、层间，还是请求级边界？
    3. 如果规模扩大一倍，通信是线性涨，还是同步点会更糟？

    能把这几个问题想清楚，很多并行策略的优缺点就不再是抽象定义，而会变成非常具体的系统判断。

    常见面试问题¶

    初级¶

    1. DP、TP、PP 分别主要通信什么？
    2. 为什么 FSDP/ZeRO 更像“切状态”而不是“切计算”？

    中级¶

    1. 为什么 TP 往往比 DP 更依赖机内高速互联？
    2. 为什么 PP 会有 bubble，即便每张卡都很忙？

    高级¶

    1. 如果模型单卡装不下，你会优先选 TP、PP 还是 FSDP？为什么？
    2. 如果 FSDP 显存降下来了但吞吐很差，你首先怀疑什么？

    易错点¶

    • 把并行策略当成互斥选项，忽略真实系统经常是组合拳
    • 只讨论显存，不讨论通信发生的频率和位置
    • 只看平均 step time，不看 rank 间是否不均衡

    排查 checklist¶

    • 当前方案切的是计算、状态，还是两者都有？
    • 最重的 collective 出现在层内、层间，还是 step 末？
    • 通信能否 overlap，还是形成了硬同步点？
    • 当前拓扑是否匹配所选并行策略？
