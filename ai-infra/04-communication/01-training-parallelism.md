# 训练并行策略：DP / TP / PP / FSDP / ZeRO

## 一句话先讲清

讨论训练并行时，最重要的不是把缩写背全，而是先回答：**你到底是在解决“装不下”的容量问题，还是在解决“跑不快”的吞吐问题。**

不同并行策略，切开的对象不同，插入的通信位置不同，对互联和同步的敏感度也完全不同。

## 为什么值得单独学

- 多卡训练的上限往往先不是 GPU 算力，而是通信量、同步点、负载均衡和内存切分方式。
- 同样是“8 卡训练”，DP 和 TP 的通信位置、频次和链路要求可能完全不一样。
- 真正决定工程可行性的，不只是“显存省了多少”，还包括：通信是否能 overlap、是否依赖高速互联、扩展后吞吐掉得有多厉害。

## 各种并行到底切了什么

| 策略 | 切分对象 | 更适合解决什么问题 | 常见代价 |
|---|---|---|---|
| DP | 切 batch | 模型能装下，但想提总吞吐 | step 末梯度同步 |
| TP | 切层内矩阵 | 单层太大，单卡算不过来或装不下 | 层内频繁通信，强依赖低延迟互联 |
| PP | 切层段 | 模型很深，适合按 stage 拆 | pipeline bubble、stage 负载均衡 |
| SP | 切序列维 | 长序列激活压力大，常与 TP 联用 | 额外序列相关通信 |
| FSDP / ZeRO | 切参数、梯度、优化器状态 | 显存主要卡在训练状态 | gather / shard 时机、通信碎片化 |

一句工程直觉：**先搞清楚“切的是什么”，再谈它省了什么、又多了什么。**

## 为什么“能跑起来”不等于“扩展得漂亮”

训练并行里最常见的误解是：

- 只要模型能分到多卡上，就应该随着卡数增加接近线性提速

现实通常不是这样，因为你还要付这些账：

- 梯度同步账
- 参数 gather / shard 账
- 激活边界传输账
- stage / rank 负载均衡账
- overlap 做没做好的账

## 一个最小例子：8 卡 DP vs 8 卡 TP

- **8 卡 DP**：每卡算不同 mini-batch，step 末做一次大梯度同步
- **8 卡 TP**：每层前后都可能同步局部结果

所以即便总卡数一样，TP 往往比 DP 更依赖低延迟高速互联，因为它把通信插进了更细的执行粒度里。

## FSDP / ZeRO 最容易被误解的地方

假设模型单卡装不下，你尝试了 FSDP：

- 显存问题解决了
- 但 step time 明显变长

这时常见根因不是 FSDP “没效果”，而是：

- 参数 gather / shard 时机不理想
- 通信没有和计算 overlap
- bucket 太小，通信被切得太碎

也就是说：**FSDP 解决的是“能不能装下”，它并不自动保证“吞吐一定更高”。**

## Troubleshooting：加卡了，为什么吞吐没按预期涨

| 现象 | 第一怀疑点 | 如何验证 |
|---|---|---|
| 卡数翻倍，吞吐涨很少 | 同步点太多或 overlap 不足 | 看 trace 中通信与计算重叠情况 |
| TP 扩展性很差 | 链路延迟或机内互联不够强 | 结合拓扑与 collective 时间看 |
| PP 利用率不高 | pipeline bubble 或 stage 不均衡 | 看各 stage 空闲时间 |
| FSDP 显存省了但更慢 | gather / shard 碎片化 | 看 bucket 设置、collective 频次 |
| 少数 rank 长期偏慢 | 负载不均或链路差异 | 看 rank 级 step time 和通信时间 |

### 一个排障顺序

1. 先分清当前瓶颈是**装不下**还是**同步太慢**。
2. 再确认切的是参数、激活、梯度还是优化器状态。
3. 然后看通信被插在执行图的哪个位置，而不是只看总通信字节数。
4. 最后结合拓扑判断：问题是策略本身，还是链路和分组方式不适配。

## 推理优化工程师视角

虽然这页主题是训练并行，但它对推理优化同样重要，因为很多系统判断用的是同一套思维方式：

- 先分清瓶颈来自计算切分，还是状态 / 通信切分。
- 先看通信插在执行图的哪个位置，而不是只看总量。
- 先判断当前互联是否支撑你想要的切分粒度。

这种思路在推理里也会复用到 TP、EP、长上下文、多机场景。真正有价值的不是背定义，而是能迅速回答：

1. 它切了什么？
2. 它会新增哪些同步点？
3. 它更怕延迟、带宽，还是拓扑不匹配？

## 选型顺序（工程直觉）

1. 模型能装进单卡但想提吞吐：先看 DP
2. 单层矩阵太大：看 TP
3. 模型太深：看 PP
4. 显存主要卡在训练状态：看 FSDP / ZeRO
5. 长上下文激活压力高：再看 SP / checkpointing

## 面试高频问法

### 初级

1. DP、TP、PP、FSDP 分别主要解决什么问题？
2. 为什么 FSDP / ZeRO 常被说成“切状态”？

### 中级

1. 为什么 TP 常比 DP 更依赖低延迟高速互联？
2. 为什么 PP 会出现 bubble？

### 高级

1. 如果模型装不下单卡，你会优先选 TP、PP 还是 FSDP？依据是什么？
2. 如果加卡后吞吐没有按预期提升，你会先看哪类通信与同步点？

## 易错点

- 只背定义，不追问“到底切了什么”
- 只关心单卡显存，不看多卡通信曲线
- 认为卡数翻倍，吞吐就应接近翻倍
- 把“能跑起来”和“扩展效率高”混为一谈

## 排查 checklist

- [ ] 当前瓶颈是装不下还是同步太慢？
- [ ] 切的是参数、激活、梯度还是优化器状态？
- [ ] profiler / trace 能看出通信占比和重叠情况吗？
- [ ] stage 或 rank 是否存在明显负载不均？
- [ ] 当前拓扑是否匹配所选并行粒度？
