# 训练并行策略：DP / TP / PP / FSDP / ZeRO

## 要点

- 多卡训练的真正上限往往不是 GPU 算力，而是 **通信量、同步点、负载均衡、内存切分方式**。
- 讨论并行时先明确：你是在解决容量问题（装不下）还是速度问题（跑不快）。
- 真正决定工程可行性的，不只是“显存省了多少”，还包括：通信发生在什么位置、是否能 overlap、是否依赖高速互联。

## 通用知识

### 它是什么

训练并行是在多张卡之间拆分训练任务。常见拆法包括：

- DP：切 batch
- TP：切层内矩阵
- PP：切层段
- SP：切序列维
- FSDP/ZeRO：切参数、梯度或优化器状态

### 它解决什么问题

- DP：模型能装下，但希望扩大 batch 和总吞吐
- TP：单层太大，单卡算不过来或装不下
- PP：模型太深，适合按层切段
- FSDP/ZeRO：显存压力主要来自训练状态

### 为什么在 AI 系统里重要

因为大模型训练里，“能不能训”通常先卡在三件事：

- 单卡显存装不装得下
- 多卡通信能不能跟上
- 扩展后吞吐掉得有多厉害

### 它的收益与代价

- 更强的切分通常能省显存或提升总吞吐
- 但也意味着更复杂的 collective、更高的同步频率和更难排查的性能问题

## 并行方式速览

- DP：每卡完整模型，梯度同步
- TP：单层矩阵切分，层内频繁通信
- PP：按层切 stage，存在 pipeline bubble
- SP：序列维切分，常与 TP 联用
- FSDP/ZeRO：切训练状态（参数/梯度/优化器）

## 通信对象与 collective

常见通信对象：

- 梯度
- 参数分片
- 激活边界张量
- 优化器状态分片

常见 collective：

- all-reduce
- all-gather
- reduce-scatter
- broadcast

## 最小例子

一个够用的直觉对照：

- 8 卡 DP：每卡算不同 mini-batch，step 末做一次大梯度同步
- 8 卡 TP：每层前后都可能要同步局部结果

所以即便两者总卡数一样，TP 往往比 DP 更依赖机内高速互联，因为它把通信插进了更细的执行粒度里。

## 工程例子

假设一个模型单卡装不下，你尝试了 FSDP：

- 显存问题解决了
- 但 step time 明显变长

这时常见根因不是 FSDP “没效果”，而是：

- 参数 gather / shard 的时机不理想
- 通信没有和计算 overlap
- bucket 太小，导致通信被切碎

也就是说，FSDP 解决的是“能不能装下”，不自动保证“吞吐一定更高”。

## 推理优化工程师视角

虽然这篇主题是训练并行，但对推理优化工程师同样重要，因为很多系统设计判断都来自同一种思维方式：

- 先分清楚瓶颈来自计算切分，还是状态/通信切分
- 先看通信被插在执行图的哪个位置，而不是只看总字节数
- 先判断当前互联是否支撑你想要的切分粒度

这种思路在推理里也会复用到 TP、EP、长上下文和多机场景中。真正有价值的不是背出 DP/TP/PP/FSDP 的定义，而是能迅速回答：

1. 它切了什么？
2. 它会多出哪些同步点？
3. 它更怕延迟、带宽，还是拓扑不匹配？

如果这三个问题说不清，后面的性能判断通常也会发虚。

## 选型顺序（工程直觉）

1. 模型能装进单卡但想提吞吐：先看 DP
2. 单层矩阵太大：看 TP
3. 模型太深：看 PP
4. 显存卡在状态：看 FSDP / ZeRO
5. 长上下文激活压力：再看 SP / checkpointing

## 常见面试问题

### 初级

1. DP、TP、PP、FSDP 分别主要解决什么问题？
2. 为什么 FSDP/ZeRO 常被说成“切状态”？

### 中级

1. 为什么 TP 常比 DP 更依赖低延迟高速互联？
2. 为什么 PP 会出现 bubble？

### 高级

1. 如果模型装不下单卡，你会优先选 TP、PP 还是 FSDP？依据是什么？
2. 如果加卡后吞吐没有按预期提升，你会先看哪类通信与同步点？

## 易错点

- 只背定义，不问“到底切了什么”
- 只关心单卡显存，不关心多卡通信曲线
- 认为卡数翻倍吞吐就应接近翻倍
- 把“能跑起来”和“扩展效率高”混为一谈

## 排查 checklist

- [ ] 当前瓶颈是装不下还是同步太慢？
- [ ] 切的是参数、激活、梯度还是优化器状态？
- [ ] profiler/trace 能看出通信占比吗？
- [ ] stage 或 rank 是否明显负载不均？
