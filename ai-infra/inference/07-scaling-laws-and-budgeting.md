# Scaling Laws 工程用法：预算、外推、实验设计

## 要点

- 从 CS336 的角度看，scaling laws 的最大价值不是“知道 Chinchilla”，而是用小规模实验来规划大规模训练预算。
- 真正的问题不是“模型越大越好”，而是：**在固定 FLOPs、固定时间、固定硬件下，模型大小 $N$ 和数据量 $D$ 怎么分配更合理**。
- 常见工作流是：
  - 先做一组小模型 isoflops 实验
  - 再拟合 loss 随模型和数据变化的趋势
  - 最后用拟合结果外推大规模预算下的最优点
- scaling laws 很有用，但不是神谕。数据质量、优化器、学习率调度、上下文长度、架构变化，都会让外推失真。

## 1. Scaling Laws 到底回答什么问题

它主要回答：

- 给定总算力预算 $C$，更该把预算放在更大的模型上，还是更多的训练 token 上？
- 小模型上的最优超参数和损失趋势，能否迁移到更大的模型？

如果没有这类工具，训练预算规划很容易变成：

- 盲目放大模型
- 或盲目堆数据
- 最后既不 compute-optimal，也不 time-optimal

## 2. 术语最小集合

- $N$：模型参数量
- $D$：训练 token 数或数据规模
- $C$：总计算预算（常用 FLOPs 表示）
- IsoFLOPs：总计算预算相同、但 $N$ 与 $D$ 分配不同的一组实验
- Compute-optimal：在给定 $C$ 下达到最低训练损失的配置

## 3. 一个够用的工程流程

### 第一步：选一组能跑得动的小实验

目标不是做最强模型，而是覆盖：

- 不同模型大小
- 不同 token 预算
- 相同或近似相同的 FLOPs 预算

这组实验的重点是：

- 可重复
- 成本可控
- 能形成趋势

### 第二步：做 IsoFLOPs 对照

也就是让多组实验在近似相同 $C$ 下，只改变：

- 模型更大、数据更少
- 或模型更小、数据更多

然后比较最终 loss 或目标指标。

### 第三步：拟合趋势

常见做法是拟合：

- loss 随 $N$ 增大如何下降
- loss 随 $D$ 增大如何下降

工程上不必执着于某个特定闭式公式的“神圣正确”，更重要的是：

- 你的拟合是否稳定
- 外推区间是否太远
- 结论是否和已有经验一致

### 第四步：决定大规模训练点

最后再回答：

- 当前预算下，下一版模型该多大
- 数据应准备多少 token
- 学习率、batch、训练时长是否也需要联动修改

## 4. 为什么大家总提 Chinchilla

因为它强调一个关键结论：

- 很多训练并不 compute-optimal
- 同样的预算下，模型大小和 token 数存在更合理的配比

一个常被引用的经验是：

- 最优数据 token 数与参数量之间存在近似固定比例

但要注意：

- 这是某类模型、某类训练设置下的经验趋势
- 不是脱离上下文就永远成立的金科玉律

## 5. 工程上最常见的误用

### 误用 1：把 scaling laws 当成论文结论背诵

真正要做的是实验设计，而不是复述名词。

### 误用 2：只拟合 loss，不看真实目标

在某些任务里，你真正关心的可能是：

- downstream accuracy
- reasoning 能力
- latency / cost 可接受性

仅看 pretraining loss，不一定足够。

### 误用 3：外推距离太远

如果小实验和目标大模型之间差了好几个数量级，拟合误差会被放大。

### 误用 4：忽略数据质量变化

scaling law 默认你在比较“同一类问题”。一旦数据分布、清洗强度、tokenizer、上下文长度都变了，趋势可能被严重污染。

## 6. 为什么这和 AI Infra 有关

很多人把 scaling laws 视为建模问题，但它其实也是基础设施问题：

- 预算 $C$ 不是抽象数字，而是 GPU 数量、训练时间、吞吐的函数
- $D$ 不是一个 token 总数而已，还涉及数据准备能力与训练输入管线能力
- 某个最优点即便数学上最好，也可能因为显存、带宽、调度限制根本不可运行

所以工程上真正合理的问题应该是：

- 在我们的硬件和时间约束下，compute-optimal 点是否真的可执行？

## 7. 一个最小预算心智模型

假设你已知：

- 每 step 处理 token 数
- 每秒 step 数或 token/s
- 总训练时长上限

则你能反推出：

- 最大可处理 token 数
- 大概能承受的模型规模
- 是否值得做更多小实验来拟合下一轮曲线

也就是说，scaling laws 的输入不是凭空来的，它建立在吞吐和资源核算之上。

## 8. 该怎么把它落到实验表里

一个够用的实验表至少应记录：

- 模型参数量
- 训练 token 数
- 最大学习率 / 调度器
- batch size / sequence length
- 总训练 FLOPs 粗估
- 最终 loss / validation loss
- 硬件与训练时间

没有这些元数据，后面的拟合与比较就会非常脆弱。

## 9. 你至少要会回答的三个问题

### 例 1：为什么大模型不一定值得继续放大

因为在固定预算下，数据不够时，继续增大模型可能落在次优区间。

### 例 2：为什么小实验不是“玩具”

因为它们不是为了最终性能，而是为了获得趋势与预算判断。

### 例 3：为什么同样的 scaling 结论在不同团队不一定复现

因为 tokenizer、数据质量、优化器、调度器、上下文长度、架构和实现细节都可能不同。

## 易错点

- 把 scaling law 只当论文结论，不把它转成实验规划工具
- 小实验记录不全，导致后续无法拟合和比较
- 只看训练 loss，不看真实目标和系统成本
- 外推时忽略实现约束，选出一个“理论最优但根本跑不动”的点

## 排查 checklist

- [ ] 你的实验表是否记录了 $N$、$D$、$C$ 及关键训练超参数？
- [ ] 你是在做 budget 规划，还是只在复述 Chinchilla 结论？
- [ ] 当前拟合的外推距离是否过远？
- [ ] 你选出的最优点，真的在现有硬件和时间预算下可执行吗？
