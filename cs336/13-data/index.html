    13｜数据工程（一）：数据来源、过滤与版权¶

    原始来源：https://tuananhbui89.github.io/blog/2025/cs336-lec13/

    先抓住这讲要点¶

    • 对 LLM 来说，数据不是配角，而是模型能力边界、行为倾向和泛化风格的第一决定因素。
    • 现代训练流程里，数据通常可以分成 pre-training → mid-training → post-training 三个阶段：规模逐步变小，但目标逐步更明确，质量要求逐步更高。
    • 数据工程绝不是“下载网页”这么简单，它同时是工程问题、统计问题、启发式问题，也是法律与合规问题。

    这一讲在整门课里的位置¶

    前面课程大量在讲模型、系统、优化、推理、评测。
    但如果数据本身质量不对，再好的模型结构和训练系统，也只是在更高效地学习错误东西。

    所以这一讲真正回答的是：

    模型最终学到的到底是谁的语言、谁的知识、谁的偏见、谁的格式？这些东西是怎么通过数据进入模型的？

    换句话说，这讲是整门课里最接近“源头控制”的一讲。

    这讲想训练你什么能力¶

    学完这一讲，你应该能：

    • 理解为什么说“数据决定行为”，而不只是“数据决定损失”；
    • 分清 pre-training、mid-training、post-training 三阶段的数据目标；
    • 看懂 Common Crawl 这类 web-scale 数据源为什么既诱人又危险；
    • 知道数据工程里为什么必须同时考虑质量、覆盖、成本、版权和发布风险。

    代表图¶

    为什么说数据决定模型行为¶

    模型不会“凭空理解世界”。
    它的知识、语气、格式、偏见、语言能力、代码能力，几乎都来自训练时看见过什么。

    这意味着下面这些问题，本质上都是数据问题：

    • 模型会不会多语言；
    • 模型会不会写代码；
    • 模型是否偏好某种文风；
    • 模型会不会更容易幻觉；
    • 模型是否对某类领域更熟；
    • 模型是否会继承有毒、有偏、低质量表达。

    所以“数据决定行为”比“数据决定 loss”更重要。
    loss 只是训练中的信号，而行为才是用户最终看到的结果。

    三阶段数据为什么目标完全不同¶

    现代大模型训练常常不是单一阶段一把梭，而是分层处理。

    1. Pre-training：覆盖与规模优先¶

    这一阶段的目标通常是：

    • 学语言统计规律；
    • 获得广泛知识覆盖；
    • 建立通用表示能力。

    所以它更强调：

    • 数据规模大；
    • 主题覆盖广；
    • 语言多样；
    • 格式丰富。

    但这也意味着这一阶段最容易混入：

    • 垃圾内容；
    • 重复内容；
    • SEO 垃圾页；
    • 版权复杂内容；
    • 风格极不均衡的网页噪声。

    2. Mid-training：能力定向¶

    这是现代模型越来越重视的一环。
    它的目标不是“再多喂一点普通网页”，而是针对性地增强某些能力，例如：

    • 数学；
    • 代码；
    • 长上下文；
    • 工具调用格式；
    • 某些特定领域知识。

    为什么它重要？
    因为在模型越来越强之后，普通通用网页继续增加的边际收益可能不如高价值定向数据。

    3. Post-training：交互、安全、偏好¶

    这一阶段关心的已经不只是“懂不懂”，而是：

    • 会不会按指令回答；
    • 会不会保持对话格式；
    • 安不安全；
    • 会不会更像用户想要的助手。

    所以 post-training 的数据量通常更小，但对：

    • 格式；
    • 质量；
    • 一致性；
    • 偏好信号；

    要求更高。

    为什么 Common Crawl 又香又难吃¶

    如果你要构建大规模语料，Common Crawl 之类的数据源几乎绕不过去。
    原因很简单：

    • 量极大；
    • 来源广；
    • 覆盖真实互联网内容；
    • 成本上相对可得。

    但问题也同样明显。原始网页里充满了：

    • boilerplate（导航栏、页脚、版权声明、模板文本）；
    • spam；
    • near-duplicate；
    • 自动生成或低价值内容；
    • 有毒文本；
    • 版权与许可风险。

    所以 Common Crawl 的价值，从来不在“原始数据包本身”，而在于：

    你能不能把海量网页垃圾，提炼成真正适合模型学习的语料。

    这也是为什么很多团队真正的壁垒，不是“有没有抓到网”，而是“有没有把网洗干净”。

    代码拆解：数据 pipeline 最小骨架¶

    def data_pipeline(raw_docs):
    docs = extract_text(raw_docs)
    docs = language_filter(docs)
    docs = deduplicate(docs)
    docs = quality_filter(docs)
    docs = license_filter(docs)
    return docs

    这段代码非常短，但几乎把大规模语料处理的骨架讲清楚了：

    1. extract_text：先把网页壳剥掉，取出真正正文；
    2. language_filter：保留目标语言或语种配比；
    3. deduplicate：防止模型一遍又一遍吃相同或近似内容；
    4. quality_filter：筛掉低质量文本；
    5. license_filter：降低合规风险。

    一句话总结就是：

    一个好数据集不是“下载下来”的，而是“加工出来”的。

    为什么数据工程不是纯算法题¶

    很多人刚接触这块时，会下意识把它理解成“训练个分类器把垃圾网页筛掉”。
    但现实远复杂得多。

    1. 它是工程题¶

    因为你面对的是：

    • TB/PB 级原始数据；
    • 分布式处理；
    • 高吞吐清洗 pipeline；
    • 存储与索引成本；
    • 多轮版本迭代。

    2. 它是启发式题¶

    因为高质量文本不总能被一个统一指标定义。
    比如：

    • 学术论文和论坛回答风格完全不同；
    • 短文本不一定低质量；
    • 模板文不一定没用；
    • 代码和自然语言的“好坏标准”不一样。

    3. 它是法律与合规题¶

    这往往是很多技术视角最容易忽略的一面。

    版权与工程现实¶

    从工程角度，你常会遇到三个不同层次的问题：

    1. 能不能爬¶

    这不只取决于“技术上能访问”，还取决于：

    • robots / ToS；
    • 平台访问规则；
    • 是否允许自动化采集。

    2. 能不能训练¶

    即便你拿到了数据，也不等于就能安全地用于模型训练。
    这里还涉及：

    • license；
    • 使用范围；
    • 再分发限制；
    • 法务风险。

    3. 能不能发布¶

    即便内部训练可接受，最终：

    • 数据能不能公开；
    • 模型是否可商用；
    • 是否需要额外披露或限制；

    又是另一层判断。

    所以 license filtering 不是“有空再做”的小步骤，而是大规模数据工程的必要防线。

    一个更成熟的数据心智模型¶

    如果把这一讲压缩成一个决策框架，可以这样记：

    1. 先明确训练阶段目标；
    2. 再找合适来源；
    3. 再做清洗、过滤和去重；
    4. 最后再做质量、合规和发布判断。

    换句话说：

    数据工作不是先拿一堆网页，再看看能不能用；而是先知道自己要教模型什么，再反推该收什么、留什么、删什么。

    面试里可以怎么讲¶

    如果面试官问：“为什么说数据决定模型行为？”

    你可以答：

    因为模型学到的知识、表达风格、语言覆盖、代码能力和偏见倾向，本质上都来自训练数据分布。模型结构决定它怎么学，数据决定它到底学到了什么。

    如果面试官问：“为什么现代模型越来越重视 mid-training？”

    可以答：

    因为在基础能力已经很强之后，继续堆通用网页数据的边际收益会下降，而针对代码、数学、长上下文等高价值能力做定向数据增强，往往更能提升实际能力和产品价值。

    复习题¶

    1. 为什么说数据比很多人想象中更直接地决定模型行为？
    2. pre-training、mid-training、post-training 三阶段的数据目标有何本质差别？
    3. Common Crawl 为什么“香”，又为什么“难吃”？
    4. 一个合格的数据 pipeline 为什么至少要有抽取、过滤、去重和 license 检查？
    5. 为什么说数据工程同时是技术、启发式和法律问题？
