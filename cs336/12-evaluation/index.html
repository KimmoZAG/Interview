    12｜评测：指标、协议与陷阱¶

    原始来源：https://tuananhbui89.github.io/blog/2025/cs336-lec12/

    先抓住这讲要点¶

    • 评测没有“脱离上下文的客观分数”，只有带协议、带分布、带目标的信号。
    • 在问“模型谁更强”之前，必须先问：你测的是 base model、prompted method，还是完整系统？
    • 输入分布、调用协议、评分标准、解释框架这四层只要变一层，分数含义就可能完全改变。

    这一讲在整门课里的位置¶

    前面课程讲了怎么训练、怎么扩展、怎么预测规模效应。
    但如果评测方式本身不可靠，那么前面很多努力都可能被错误解读。

    这一讲真正要回答的是：

    当我们说“这个模型更强”时，到底是在说什么？这个“强”是在哪种任务定义、哪种输入协议、哪种评分机制下成立的？

    所以评测并不是训练之后附带的一步，而是贯穿研究、工程和产品决策的解释框架。

    这讲想训练你什么能力¶

    学完这一讲，你应该能：

    • 区分“分数高”和“能力更强”之间的差异；
    • 读 benchmark 结果时主动追问协议和上下文；
    • 理解 perplexity、accuracy、preference win-rate 等指标各自适用的范围；
    • 对开放式任务知道为什么评测这么难，以及常见替代方案的偏差来源。

    代表图¶

    为什么 benchmark 分高，不一定代表“更强”¶

    这是大模型评测里最容易被忽略的常识。
    一个模型在某 benchmark 上高几分，可能有很多解释：

    • prompt 写得更好；
    • 在该 benchmark 上污染更严重；
    • 带了 CoT、self-consistency、检索、工具；
    • 计算预算更高；
    • 牺牲了延迟、成本、稳定性。

    所以如果只报一个数字，不交代配置，你根本不知道自己在比较什么。

    更准确地说，评测至少可能在比较三种不同对象：

    1. 裸模型（model）；
    2. 调用方法（method / prompting pipeline）；
    3. 完整系统（system）。

    这三者混在一起时，很多排行榜就会变成“苹果、梨、榴莲一起比甜度”。

    评测的第一原则：先问“谁会用这个结果”¶

    评测不是为评测而评测。
    如果你不知道结果服务谁，就很难决定该测什么。

    1. 研究视角¶

    研究者更关心：

    • 模型能力是否有真实提升；
    • 某种训练方法是否有效；
    • 某种归纳偏置或数据策略是否带来泛化收益。

    2. 工程视角¶

    工程团队更关心：

    • 同样成本下哪个方案更稳；
    • 是否能支撑目标延迟和吞吐；
    • 是否容易集成到现有系统；
    • 是否在长尾 case 上可靠。

    3. 产品视角¶

    产品团队更关心：

    • 用户感知是否更好；
    • 特定使用场景里是否更有帮助；
    • 是否降低失败率、投诉率或人工介入率。

    所以一套好的评测，第一步不是选 benchmark，而是先明确“决策对象是谁”。

    四层框架：为什么同一个分数可以有完全不同含义¶

    可以把评测拆成四层来看。

    1. 输入分布（input distribution）¶

    你喂给模型的是什么：

    • MMLU 这种学术问答；
    • 真实客服工单；
    • 编程任务；
    • 多轮 agent 轨迹；
    • 安全红队样本。

    输入分布变了，结论往往也就变了。

    2. 调用协议（invocation protocol）¶

    是 zero-shot、few-shot、CoT、tool-use，还是带检索？
    同一个模型，不同调用协议表现可能差异巨大。

    3. 评分方式（metric / judging）¶

    是 accuracy、F1、pass@k、win-rate、human preference，还是规则校验？
    不同指标衡量的是不同侧面。

    4. 解释框架（interpretation）¶

    拿到分数后，你打算怎么解释？
    是说“模型逻辑推理更强了”，还是只说“在这套协议下得分更高”？

    很多争议其实不发生在分数本身，而发生在过度解释上。

    perplexity 为什么仍然重要，但远远不够¶

    在大模型时代，perplexity 经常被嫌“太 old school”。
    但它其实仍然非常有价值。

    它适合做什么¶

    perplexity 是连续、细粒度、样本效率高的信号，很适合：

    • scaling law 拟合；
    • 训练过程监控；
    • 数据配比比较；
    • 小改动的快速筛选。

    它不适合替代什么¶

    但它不能自动代表：

    • instruction following；
    • 多轮交互体验；
    • tool use；
    • agent 规划能力；
    • 安全性与可靠性。

    所以最合理的态度不是“perplexity 没用了”，而是：

    它是底层训练信号，不是全能能力代理。

    开放式评测为什么格外难¶

    像数学选择题、闭卷 QA 这类任务，至少还有相对明确的 gold answer。
    但开放式生成常常没有唯一正确答案。

    这时你就必须借助其他判断机制：

    • human preference；
    • model-as-a-judge；
    • rule-based constraints；
    • task-specific verifier；
    • downstream task success signal。

    每一种方案都有偏差。

    human preference 的问题¶

    • 成本高；
    • 标注者一致性有限；
    • 容易被文风、长度、措辞影响。

    model-as-a-judge 的问题¶

    • judge 自己有偏差；
    • 容易偏爱某些表达风格；
    • 可能和被测模型共享污染或偏见。

    rule-based / verifier 的问题¶

    • 只适用于可验证任务；
    • 很容易漏掉“看起来对、其实错”的复杂情况；
    • 覆盖范围有限。

    所以开放式评测从来不是“找一个万能 judge”，而是组合不同信号，尽量减少误判。

    代码拆解：一个评测配置至少该包含什么¶

    eval_config = {
    "input_distribution": "mmlu_zero_shot",
    "invocation": "direct_answer",
    "tool_use": False,
    "metric": "accuracy",
    "decontamination": True,
}

    这段配置虽然短，但它表达了一个非常成熟的观念：

    没有协议说明的分数，基本不值得信。

    为什么？因为每个字段都会改变结果含义：

    • input_distribution 说明你到底测的是什么任务分布；
    • invocation 说明模型是怎么被调用的；
    • tool_use 说明这是不是纯模型能力；
    • metric 决定你把什么当成“好”；
    • decontamination 决定分数是否可能被数据污染抬高。

    换句话说，评测配置不是附录，而是结果解释的一部分。

    数据污染为什么会毁掉 benchmark¶

    如果训练数据里已经出现了 benchmark 题目或非常接近的变体，那么高分就不一定代表真正泛化。
    它可能只是：

    • 记住了答案；
    • 记住了题型模式；
    • 在近重复样本上表现异常好。

    这就是为什么严肃评测里，去污染（decontamination）非常重要。
    否则你以为自己在测能力，实际上在测记忆残留。

    一个更成熟的评测心智模型¶

    如果你以后做模型评测，可以优先用下面这套顺序思考：

    1. 明确你要服务的决策目标；
    2. 选与目标匹配的输入分布；
    3. 固定调用协议；
    4. 明确评分规则；
    5. 检查是否存在污染；
    6. 再决定怎么解释结果。

    如果其中任何一步没说清楚，结论就容易被滥用。

    面试里可以怎么讲¶

    如果面试官问：“为什么说 benchmark score 是 context-dependent signal？”

    你可以答：

    因为同一个模型分数会受到输入分布、prompt 协议、是否使用工具、评分方法以及去污染处理等多种因素影响。脱离这些上下文，单个分数并不能唯一代表模型真实能力。

    如果面试官问：“perplexity 为什么重要但不够？”

    可以答：

    perplexity 是非常好的训练和 scaling 信号，适合监控优化过程、比较数据混配和做小规模实验筛选，但它不能完整代表 instruction following、多轮交互、agent 能力和安全性，所以必须和任务型评测结合使用。

    复习题¶

    1. 为什么 benchmark 高分不一定能推出“模型更强”？
    2. 评测里“模型”“方法”“系统”三者为什么必须区分？
    3. perplexity 最适合哪类分析，最不适合替代哪类能力判断？
    4. 开放式任务为什么很难找到完全无偏的评测方式？
    5. 为什么 invocation protocol 和 decontamination 必须写进评测说明？
