# 07｜并行训练（一）：策略全景

原始来源：<https://tuananhbui89.github.io/blog/2025/cs336-lec07/>

## 先抓住这讲要点

- 大模型训练里的“并行”不是单个技巧，而是一套**围绕显存、算力、通信带宽做资源重分配**的方法论。
- DDP 解决“怎么把样本分出去”；ZeRO / FSDP 解决“为什么每卡都要存完整状态”；TP / PP 解决“单卡或单节点根本装不下”；SP / activation checkpointing 解决“激活太大”。
- 真正会做系统设计的人，不是会背缩写，而是能回答：**这个并行策略省了什么、又引入了什么通信代价**。

## 这一讲在整门课里的位置

前面第 5、6 讲已经告诉我们：单卡性能有天花板，很多瓶颈来自 memory 和 kernel。  
但当模型规模继续上去，光把单卡打磨到极致还不够，因为很快会遇到两个更残酷的问题：

1. **模型放不下**；
2. **训练速度太慢**。

所以这讲真正回答的是：

> 当单卡已经不是世界的中心时，我们如何把一整个训练任务拆到很多卡上，同时尽量不把收益都赔在通信上？

## 这讲想训练你什么能力

如果你是工程师，这讲训练的是一种“资源账本意识”：

- 你能不能判断瓶颈是参数、梯度、优化器状态，还是激活？
- 你能不能判断当前更应该上 DDP、FSDP，还是 TP / PP？
- 你能不能在吞吐、实现复杂度、稳定性之间做取舍？

一句话：**并行训练不是技术名词集合，而是部署约束下的架构设计题。**

## 从一个最朴素的问题开始：为什么单卡不够？

假设你有一个 70B 模型，哪怕只看最基本的参数量，单精度、半精度、优化器状态、梯度、激活，一层层加起来，显存账目会迅速爆炸。  
于是你会发现，所谓“扩到多卡”，其实不是为了炫酷，而是因为：

- **参数**可能放不下；
- **优化器状态**更放不下；
- **激活**在长序列和大 batch 下也放不下；
- 即便放得下，训练时间也长到难以接受。

因此，并行训练本质上是在做两件事：

1. 把存储压力拆开；
2. 把计算工作拆开。

只是这两件事都会带来第三件事：

3. **通信压力暴涨。**

## 并行训练的总地图：你到底在切什么

把所有策略放在一起看，最清楚的方式是问一句：**你切的对象是什么？**

### 1. Data Parallelism：切样本

每张卡都有完整模型，但每张卡看不同 mini-batch 切片。  
这是最自然、最稳定、最容易起步的方案。

它的优点是：

- 编程模型简单；
- 单卡计算逻辑几乎不变；
- 很适合做“先把训练跑起来”的第一版系统。

它的缺点也非常明显：

- 参数每卡一份；
- 梯度每卡一份；
- optimizer state 每卡也一份；
- 到了超大模型时代，这种“每卡都要全家桶”的模式非常贵。

### 2. Tensor Parallelism：切一层内部的算子

不是每张卡算不同样本，而是**多张卡一起算同一层**。  
比如一个超大的线性层，按列切、按行切，让不同 GPU 各算一部分输出或输入贡献。

它解决的是：

- 单层太大，一张卡算不过来；
- 需要把单层矩阵乘的参数和计算都拆开。

它的代价是：

- 层内通信频繁；
- 对低延迟、高带宽互联很敏感；
- 通常更适合单节点内用 NVLink / NVSwitch 托底。

### 3. Pipeline Parallelism：切层

把模型前几层放 A 卡，中间几层放 B 卡，后几层放 C 卡……  
本质上是把网络深度方向分段。

它解决的是：

- 模型整体放不下；
- 不希望层内切分太细；
- 想跨节点扩展更大的模型。

它的核心挑战是“流水线气泡”：

- 前面 stage 还没把 micro-batch 送来，后面 stage 就在等；
- micro-batch 太少，流水线吃不满；
- 负载不均衡会让某个 stage 成为全局瓶颈。

### 4. Sharding（ZeRO / FSDP）：切训练状态

这个思路很“会过日子”：

> 既然很多状态不是时时刻刻都要被每张卡完整持有，那为什么不把它们分片存呢？

这类方法重点不是把前向计算改得多花哨，而是把：

- optimizer state
- gradients
- parameters

从“每卡全拷贝”变成“跨卡分片 + 按需聚合”。

### 5. Sequence / Activation 并行：切中间结果

当序列长、batch 大、层数深时，激活是隐藏的大户。  
所以还有一类策略专门围绕激活做文章，比如：

- activation checkpointing / recomputation
- sequence parallelism
- selective activation offload

它们关心的是：

> 我能不能少存一点中间结果，等需要的时候再算回来？

这是一种典型的“**用计算换显存**”的思路。

## DDP：最经典，也最容易踩上天花板

### DDP 的工作方式

DDP 的流程可以用一句话概括：

> 本地前向、本地反向、最后把梯度同步起来。

也就是说，每张卡：

1. 拿一份完整模型；
2. 算自己那部分数据；
3. 得到本地梯度；
4. 对梯度做 `all-reduce`；
5. 各卡用相同梯度更新，所以参数继续保持一致。

## 代码拆解：all-reduce 视角下的 DDP

```python
# 伪代码
for batch in dataloader:
    loss = model(batch)
    loss.backward()          # 每卡得到本地梯度
    all_reduce(grads)        # 求和/平均
    optimizer.step()         # 每卡更新后参数保持一致
```

这段代码看着朴素，但它已经揭示了 DDP 的全部气质：

- 计算几乎完全本地化；
- 同步只发生在梯度阶段；
- 所以实现简单、鲁棒性高；
- 但内存复制问题一点没解决。

### DDP 的主要瓶颈是什么

1. **状态复制严重**：参数、梯度、优化器状态都要每卡持有；
2. **梯度同步成本高**：模型越大，反向后 all-reduce 越重；
3. **扩卡收益受 global batch 约束**：如果 batch 不够大，扩卡后每卡样本过少，算力利用会下滑；
4. **通信与计算 overlap 不总是理想**：桶化（bucketing）做不好，尾部同步会很重。

所以 DDP 最大的优点是“简单”，最大的问题也恰恰是“太诚实”：  
它几乎没有隐藏任何成本。

## ZeRO / FSDP：从“整卡复制”走向“按需聚合”

如果说 DDP 的问题是“每卡都放整套”，那 ZeRO / FSDP 的核心思想就是：

> 把能拆的都拆开，别让每个 GPU 做冤大头。

### ZeRO-1 / 2 / 3 在分什么

- **ZeRO-1**：只分 optimizer state；
- **ZeRO-2**：在 ZeRO-1 基础上再分 gradients；
- **ZeRO-3**：连 parameters 也分；
- **FSDP**：本质上是参数分片 + 按层或按模块 all-gather / reshard 的工程实现路线。

你可以把它理解成一个渐进过程：

1. 先承认 optimizer state 太大；
2. 再承认 gradients 也很大；
3. 最后承认 parameters 本身也不能每卡全放。

### FSDP 的直觉为什么重要

FSDP 最有代表性的动作是：

1. 某层要算之前，先把这一层参数从 shards 聚齐；
2. 用完整参数完成该层前向 / 反向；
3. 用完后再把完整参数释放，只保留分片。

## 代码拆解：FSDP 的层级直觉

```python
# 伪代码
for layer in model.layers:
    full_param = all_gather(layer.shard)
    out = layer.forward(x, full_param)
    free(full_param)
```

这段伪代码看似简单，却非常传神：

- **参数不是常驻完整态，而是按需临时拼起来**；
- 省下来的显存，换来的是更碎片化、更频繁的通信；
- 工程上你还得认真处理 prefetch、bucket、overlap、wrap 粒度等问题。

一句话总结：

> FSDP 不是白嫖显存，而是拿通信调度复杂度去换显存容量。

## TP 和 PP：一个切算子，一个切网络深度

### Tensor Parallelism 的适用场景

TP 常用于：

- hidden size 很大；
- 单层 GEMM 特别大；
- 单机内互联足够快；
- 希望同一层并行展开。

典型例子是把一个线性层按列切开：

```python
def column_parallel_linear(x, weight_shards):
    partial_outputs = [x @ w for w in weight_shards]
    return concat(partial_outputs, dim=-1)
```

它的核心特点是：**层内每一步都可能需要 collective**。  
所以 TP 性能很依赖拓扑，不是“卡多就赢”。

### Pipeline Parallelism 的适用场景

PP 常用于：

- 模型很深；
- 单层切分不方便或不想切太细；
- 想跨更多节点扩展；
- 愿意通过 micro-batch 流水线换可扩展性。

一个朴素的流水线可以想成：

```python
stages = [embed_and_block1, block2_to_20, block21_to_40, lm_head]

for micro_batch in micro_batches:
    x = micro_batch
    for stage in stages:
        x = stage(x)
```

现实中的关键不是“能不能过一遍”，而是：

- stage 是否均衡；
- micro-batch 数够不够；
- bubble 占比是否太高；
- activation 传输和 recompute 怎么配合。

## 为什么现实里几乎总是混搭

真正的大模型训练配置，往往不是“只用一种并行”，而是这样的组合题：

- 数据并行负责扩大 batch；
- FSDP / ZeRO 负责控制状态内存；
- TP 负责切大层；
- PP 负责跨设备容纳更多层；
- activation checkpointing 负责把激活压住。

所以并行系统设计常见的表达方式不是“我用了某个技术”，而是：

> 我在 `DP × TP × PP × shard × recompute` 的组合空间里，找到了当前机器、预算、模型最合适的那个点。

## 选型时真正该问的问题

很多同学学完并行训练后，脑中只剩一个表格：这个省参数，那个省激活。  
但工程里更重要的问题其实是下面这些：

### 1. 你现在首先缺什么

- 显存不够？
- 单层太大？
- 机器间带宽不够？
- 训练吞吐太低？
- global batch 想拉大？

不同瓶颈，首选方案完全不同。

### 2. 你的机器拓扑长什么样

- 单节点 8 卡 NVLink？
- 多节点 IB，但节点内外带宽差很多？
- 是否有 NVSwitch？

拓扑决定 TP 是否友好，也决定 FSDP / PP 的通信组织方式是否划算。

### 3. 你的团队能承受多高的系统复杂度

DDP 最简单。  
FSDP、TP、PP 越往后，系统复杂度、调试难度、hang 风险、调参难度通常越高。  
不是不能上，而是要算**工程人力账**。

## 面试里可以怎么讲

如果面试官问：**“并行训练策略怎么选？”**

你可以这样回答：

> 我会先按瓶颈拆问题。若模型能放下但想扩 batch，优先 DDP；若状态内存太大，优先 ZeRO / FSDP；若单层算子太大且节点内互联强，优先 TP；若模型深度过深或要跨节点扩展，考虑 PP。真实系统中通常是混合并行，因为每种策略解决的是不同维度的资源瓶颈。

如果面试官继续追问：**“为什么 TP 往往放单节点内？”**

一个很强的回答是：

> 因为 TP 的同步发生在层内前后向关键路径上，频率高、延迟敏感，所以更适合部署在 NVLink / NVSwitch 这类高带宽低延迟互联内部；跨节点做 TP，往往通信代价会侵蚀收益。

## 复习题

1. DDP 为什么简单，但又为什么很快遇到显存天花板？
2. ZeRO-1 / 2 / 3 分别在减少哪类复制？
3. 为什么说 FSDP 是“按需聚合、用完即散”？
4. TP 和 PP 分别在切什么，适用拓扑有何不同？
5. 如果你的问题是“激活爆了”，那你首先会考虑哪些手段？
