- 学大模型不能只停留在 API 调用层，必须把**模型、数据、系统**三个栈串起来。
- 很多结论都带强烈的**规模依赖**：小模型上看到的现象，不一定能外推到大模型。
- Tokenizer 是训练成本的第一层压缩器：token 越少，后面的 attention、KV cache、训练吞吐压力越小。
- CS336 这门课的核心精神是：**从第一性原理理解语言模型，并始终带着资源效率意识做设计。**

## 中文解读

### 1. 为什么要“从零构建语言模型”

课程强调的不是会调接口，而是要理解：

- 一条文本怎么变成 token；
- token 怎么进 Transformer；
- 前向/反向到底在算什么；
- 参数、激活、通信为什么会变成瓶颈；
- 为什么某些 trick 在大规模时才显得重要。

一句话总结：

> 如果你只会调 API，那么你是在“使用模型”；如果你能拆开 tokenizer、training loop、kernel、parallelism，你才是在“研究和构建模型”。

### 2. 工业化为什么让“从头理解”更重要

现在最强的模型往往：

- 权重不开源；
- 数据配方不公开；
- 训练 pipeline 不透明；
- 系统优化细节只公开一部分。

这意味着你不能指望通过“照抄一个完整配方”掌握模型能力。  
你必须有能力把公开论文、零散工程经验、以及可复现实验拼接起来。

也正因为如此，CS336 才会特别强调：

- 小规模可复现实验；
- 对资源的精确核算；
- 通过 scaling law 或系统分析去推断大规模行为。

### 3. 为什么说很多经验都带“规模依赖”

模型越大，很多现象都会变：

- attention 和 MLP 的成本占比会变；
- 某些优化 trick 到大模型才显著；
- 一些小模型上看起来很聪明的改法，到大模型可能没意义；
- 某些能力会出现“阈值式跃迁”。

所以这门课一直在提醒你：

> 小实验很重要，但它们不是终点；小实验的意义，在于帮助你推断大实验，而不是替代大实验。

### 4. Tokenization 为什么不是“小细节”

如果我们把一段文本压缩得更紧，那么同样语义内容对应的 token 序列会更短。  
而 Transformer 的很多核心代价，都和序列长度直接相关。

例如 self-attention 的典型复杂度近似是：

$$
O(n^2)
$$

这意味着：

- token 数减半，attention 核心代价可能降到四分之一；
- KV cache 的增长速度也会变慢；
- 推理时每次 decode 要访问的历史状态也会减少。

所以 tokenizer 不是“训练前顺手做一下”的流程，它经常是后面系统效率的起点。

## 为什么 primitive tokenizer 都不完美

### 字符级 / byte 级

优点：

- 通用；
- 可逆；
- 不会 OOV。

缺点：

- 序列太长；
- attention 成本高；
- 一个有明确语义的词，可能会被切成很多碎片。

### 词级

优点：

- 常见词压得很短；
- 序列长度好看。

缺点：

- 词表巨大；
- 稀有词、拼写变体、多语言混合很麻烦；
- 可扩展性差。

这就催生了 BPE 这种折中方案：

- 保留较短序列；
- 维持可控词表；
- 又不至于像词级那样脆弱。

## BPE 的核心思想

BPE 的思路非常朴素：

1. 从 byte 或字符开始；
2. 统计最常见的相邻 token 对；
3. 把最频繁的一对合并成新 token；
4. 重复多轮，得到词表和 merge 规则。

它在三件事中找了个平衡：

- 词表不能无限大；
- 序列不能太长；
- 编码解码要可逆。

换句话说，BPE 是一种**按数据分布分配词表容量**的方法：

- 高频子串，值得拥有独立 token；
- 低频字符串，不值得浪费一个完整词条，就拆成更小单元。

## 代码拆解：最小 BPE 思路

```python
from collections import Counter

def count_pairs(tokens):
    pairs = Counter()
    for seq in tokens:
        for i in range(len(seq) - 1):
            pairs[(seq[i], seq[i + 1])] += 1
    return pairs

def merge_pair(tokens, pair):
    a, b = pair
    merged = []
    for seq in tokens:
        out, i = [], 0
        while i < len(seq):
            if i + 1 < len(seq) and seq[i] == a and seq[i + 1] == b:
                out.append(a + b)
                i += 2
            else:
                out.append(seq[i])
                i += 1
        merged.append(out)
    return merged
```

这段代码的工程含义是：**BPE 本质上就是“频繁对子统计 + 局部替换”**。

### 这段代码真正对应的工程问题

这段代码虽然短，但已经暴露了线上 tokenizer 实现最关心的问题：

1. **pair 计数怎么高效维护**  
   朴素做法每轮全量重扫，复杂度很高。

2. **merge 怎么快速应用**  
   真实系统通常不会按所有 merge 规则一条条回放，而是尽量只处理和当前输入有关的规则。

3. **预切分（pre-tokenization）怎么做**  
   空格、标点、特殊 token 的处理会影响编码质量与可逆性。

### 一个极简训练例子

假设训练语料只有：

```text
low
lower
lowest
```

如果一开始按字符切，可能会多次看到：

- `l + o`
- `lo + w`
- `low + e`

那么 BPE 就会逐渐把高频子串组合成：

- `lo`
- `low`
- `er`
- `est`

这件事的本质并不是“造新词”，而是：

> 把重复出现的局部模式压缩成更短的记号，以节省后续模型处理长度。

## 工程视角：Tokenizer 影响哪些下游成本

你可以把 tokenizer 看成整个 LM 系统的第一个杠杆。它会影响：

### 1. 训练成本

同样一份数据，token 越少：

- 每个 batch 的序列更短；
- attention 更便宜；
- 激活更省；
- 通信量也可能下降。

### 2. 推理成本

尤其在长上下文推理里：

- token 数越多，KV cache 越大；
- decode 每步读取的历史越长；
- TTFT 和吞吐都可能变差。

### 3. 多语言表现

词表过小，某些语言会被切得很碎；  
词表过大，又会带来 embedding 成本、训练数据稀疏性和部署包体问题。

### 4. 产品表现

用户往往按 token 计费、按 token 限长。  
所以 tokenizer 其实还会影响：

- 成本体验；
- 长文输入能力；
- 多语言产品一致性。

## 面试里可以怎么讲这一讲

如果面试官问：**“为什么 tokenizer 很重要？”**

你可以用下面这版答法：

> Tokenizer 不只是文本预处理，它本质上在决定模型看到的离散单元是什么。这个选择会同时影响语义压缩率、序列长度、attention 复杂度、KV cache 大小和多语言表现。像 BPE 这种方法，本质上是在词表大小和序列长度之间做数据驱动的折中，所以它会对训练成本和推理成本都有直接影响。

如果面试官问：**“为什么要从头学语言模型，而不是直接调开源框架？”**

你可以答：

> 因为很多决定模型效果的关键因素不在 API 层，而在 tokenizer、训练 loop、资源核算、并行策略和数据 pipeline 这些底层机制上。只有理解这些，才能判断一个结论到底是架构贡献、数据贡献，还是系统实现带来的假象。

## 本讲小结

这一讲真正想建立的，不是 tokenizer 的算法细节，而是一种整体观：

- 模型效果来自整条链路；
- 资源效率必须和能力一起考虑；
- tokenizer 是大模型系统里最前面的“压缩与表达接口”。

后面所有主题——Transformer、GPU、并行、推理、数据、对齐——其实都能回到这套思路上继续展开。
