    01｜课程总览与 Tokenization¶

    原始来源：https://tuananhbui89.github.io/blog/2025/cs336-lec01/

    课程导航：课程索引 00-index｜学习路线 study-roadmap｜面试指南 interview-prep-guide｜下一讲 02 PyTorch 与资源核算

    先抓住这讲要点¶

    • 学大模型不能只停留在 API 调用层，必须把模型、数据、系统三个栈串起来。
    • 很多结论都带强烈的规模依赖：小模型上看到的现象，不一定能外推到大模型。
    • Tokenizer 是训练成本的第一层压缩器：token 越少，后面的 attention、KV cache、训练吞吐压力越小。
    • CS336 这门课的核心精神是：从第一性原理理解语言模型，并始终带着资源效率意识做设计。

    代表图¶

    这讲在整门课里的位置¶

    这讲其实在回答三个非常根本的问题：

    1. 为什么要从零理解语言模型？
    2. 为什么“效率”比单纯追求“更大”更重要？
    3. 为什么 tokenizer 看起来像前处理，实际上却深刻影响整个训练与推理系统？

    如果你把后面 17 讲看成一条完整流水线，那么这一讲是在给整门课定世界观：

    • 语言模型不是纯算法问题；
    • 也不是纯工程问题；
    • 它是模型设计、数据质量、系统约束共同作用的结果。

    中文解读¶

    1. 为什么要“从零构建语言模型”¶

    课程强调的不是会调接口，而是要理解：

    • 一条文本怎么变成 token；
    • token 怎么进 Transformer；
    • 前向/反向到底在算什么；
    • 参数、激活、通信为什么会变成瓶颈；
    • 为什么某些 trick 在大规模时才显得重要。

    一句话总结：

    如果你只会调 API，那么你是在“使用模型”；如果你能拆开 tokenizer、training loop、kernel、parallelism，你才是在“研究和构建模型”。

    2. 工业化为什么让“从头理解”更重要¶

    现在最强的模型往往：

    • 权重不开源；
    • 数据配方不公开；
    • 训练 pipeline 不透明；
    • 系统优化细节只公开一部分。

    这意味着你不能指望通过“照抄一个完整配方”掌握模型能力。
    你必须有能力把公开论文、零散工程经验、以及可复现实验拼接起来。

    也正因为如此，CS336 才会特别强调：

    • 小规模可复现实验；
    • 对资源的精确核算；
    • 通过 scaling law 或系统分析去推断大规模行为。

    这里其实隐含着一个很重要的方法论：

    • 你未必能拿到 frontier 模型的完整训练配方；
    • 但你仍然可以通过公开材料、可复现实验、资源建模，把很多关键结论重新拼出来；
    • 真正有价值的能力，不是“背到某个闭源模型的参数表”，而是看到现象后，能判断它来自模型、数据还是系统瓶颈。

    这也是为什么课程一开始就把 tokenizer 放在最前面。它不是因为简单，而是因为它足够基础：你一旦理解了“文本如何被离散化”，后面很多关于算力、显存、吞吐、长上下文的问题都会开始变得具体。

    3. 这门课其实在训练哪三种能力¶

    结合课程原始材料，可以把它总结成三类：

    1. Mechanics（机制）：真的知道一层层在算什么。
      比如 tokenizer 如何编码，attention 如何做矩阵乘，optimizer 如何更新参数，parallelism 如何切图。

    2. Mindset（方法）：始终带着效率意识和资源约束去看问题。
      不是只问“这样能不能更准”，还要问“每多 1% 效果要多花多少 FLOPs、显存、通信和工程复杂度”。

    3. Intuition（经验直觉）：知道哪些经验值得信，哪些经验可能只在某个规模区间成立。
      这部分不能只靠看论文摘要获得，很多时候只能通过实验、复现和失败慢慢建立。

    把这三者放在一起看，会更容易理解为什么课程反复强调“自己实现、自己 benchmark、自己做资源核算”。因为大模型工程里，最危险的不是“不知道答案”，而是拿着局部经验当普遍规律。

    4. 为什么说很多经验都带“规模依赖”¶

    模型越大，很多现象都会变：

    • attention 和 MLP 的成本占比会变；
    • 某些优化 trick 到大模型才显著；
    • 一些小模型上看起来很聪明的改法，到大模型可能没意义；
    • 某些能力会出现“阈值式跃迁”。

    所以这门课一直在提醒你：

    小实验很重要，但它们不是终点；小实验的意义，在于帮助你推断大实验，而不是替代大实验。

    更具体地说，规模依赖常常来自下面几个原因：

    • 计算瓶颈会迁移：小模型时你可能主要受 kernel launch、框架开销影响；大模型时则可能主要受 HBM 带宽、跨机通信或 KV cache 限制。
    • 误差来源会变：小模型往往更容易 underfit，大模型则更容易暴露数据质量、采样策略、对齐目标等问题。
    • 收益函数会变形：某个 trick 在小模型上几乎没用，可能因为主瓶颈根本不在那里；一旦规模变大，这个 trick 才开始显著改善吞吐或稳定性。
    • 能力出现阈值效应：某些推理、多步规划、长程依赖能力，不是线性增长，而是跨过某个参数量或训练 token 阈值后才明显出现。

    所以在做实验设计时，一个比较成熟的表述不是“这个方法有效”，而是：

    这个方法在什么模型规模、数据规模、训练预算、上下文长度下有效。

    5. Tokenization 为什么不是“小细节”¶

    如果我们把一段文本压缩得更紧，那么同样语义内容对应的 token 序列会更短。
    而 Transformer 的很多核心代价，都和序列长度直接相关。

    例如 self-attention 的典型复杂度近似是：

    \[ O(n^2) \]

    这意味着：

    • token 数减半，attention 核心代价可能降到四分之一；
    • KV cache 的增长速度也会变慢；
    • 推理时每次 decode 要访问的历史状态也会减少。

    所以 tokenizer 不是“训练前顺手做一下”的流程，它经常是后面系统效率的起点。

    如果把这件事再说得更工程一点，可以把文本处理链路想成：

    原始文本 -> normalization -> pre-tokenization -> subword merges -> token ids

    这里每一层都在影响后面的成本：

    • normalization 决定很多表面不同、底层相同的字符串要不要折叠到一起；
    • pre-tokenization 决定哪些边界不允许 merge 跨过去，比如空格、标点、数字；
    • subword merges 决定高频模式如何被压缩成更短的 token；
    • token ids 最终决定模型训练时真正处理的序列长度。

    所以 tokenizer 从来不是一个单独算法名词，它更像一条完整的文本编码流水线。

    为什么 primitive tokenizer 都不完美¶

    字符级 / byte 级¶

    优点：

    • 通用；
    • 可逆；
    • 不会 OOV。

    缺点：

    • 序列太长；
    • attention 成本高；
    • 一个有明确语义的词，可能会被切成很多碎片。

    词级¶

    优点：

    • 常见词压得很短；
    • 序列长度好看。

    缺点：

    • 词表巨大；
    • 稀有词、拼写变体、多语言混合很麻烦；
    • 可扩展性差。

    这就催生了 BPE 这种折中方案：

    • 保留较短序列；
    • 维持可控词表；
    • 又不至于像词级那样脆弱。

    还有一个很实用的视角：

    • 字符级 / byte 级 更像“保证覆盖率”；
    • 词级 更像“极端追求压缩率”；
    • BPE / 子词级 则是在“覆盖率、压缩率、词表大小”之间找工程上最稳妥的平衡点。

    这也是为什么今天大量 LLM 仍然使用 BPE、WordPiece、Unigram 这类子词方法，而不是简单回到纯词级或纯字符级。

    BPE 的核心思想¶

    BPE 的思路非常朴素：

    1. 从 byte 或字符开始；
    2. 统计最常见的相邻 token 对；
    3. 把最频繁的一对合并成新 token；
    4. 重复多轮，得到词表和 merge 规则。

    它在三件事中找了个平衡：

    • 词表不能无限大；
    • 序列不能太长；
    • 编码解码要可逆。

    换句话说，BPE 是一种按数据分布分配词表容量的方法：

    • 高频子串，值得拥有独立 token；
    • 低频字符串，不值得浪费一个完整词条，就拆成更小单元。

    如果从压缩的角度理解，BPE 做的是一件很“贪心”的事：

    • 哪些局部模式在语料里反复出现；
    • 哪些模式一旦被压成一个 token，就能立刻减少最多的总长度；
    • 就优先把这些模式吸进词表。

    它并不保证全局最优，但它在工程上有三个非常大的优点：

    • 训练规则简单；
    • 编码结果可解释；
    • 生成的词表通常足够好用，而且易于部署。

    byte-level BPE 为什么在大模型里很常见¶

    很多现代 LLM 并不是从 Unicode 字符直接起步，而是从 byte 起步做 BPE。原因非常现实：

    • byte 基础词表大小固定为 256；
    • 任何文本最终都能表示成 byte 序列，因此几乎不会出现真正意义上的 OOV；
    • 对混合语言、特殊字符、emoji、代码符号更稳健。

    它的代价是：

    • 中间表示可读性很差；
    • 非 ASCII 文本会被映射成不直观的字节模式；
    • 如果预切分和空格规则设计得不好，token 行为会让人觉得“反直觉”。

    但从训练系统角度看，可覆盖全部输入而又保持词表有界，通常比“人眼看起来是否优雅”更重要。

    代码拆解：最小 BPE 思路¶

    from collections import Counter

def count_pairs(tokens):
    pairs = Counter()
    for seq in tokens:
        for i in range(len(seq) - 1):
            pairs[(seq[i], seq[i + 1])] += 1
    return pairs

def merge_pair(tokens, pair):
    a, b = pair
    merged = []
    for seq in tokens:
        out, i = [], 0
        while i < len(seq):
            if i + 1 < len(seq) and seq[i] == a and seq[i + 1] == b:
                out.append(a + b)
                i += 2
            else:
                out.append(seq[i])
                i += 1
        merged.append(out)
    return merged

    这段代码的工程含义是：BPE 本质上就是“频繁对子统计 + 局部替换”。

    这段代码真正对应的工程问题¶

    这段代码虽然短，但已经暴露了线上 tokenizer 实现最关心的问题：

    1. pair 计数怎么高效维护
      朴素做法每轮全量重扫，复杂度很高。

    2. merge 怎么快速应用
      真实系统通常不会按所有 merge 规则一条条回放，而是尽量只处理和当前输入有关的规则。

    3. 预切分（pre-tokenization）怎么做
      空格、标点、特殊 token 的处理会影响编码质量与可逆性。

    如果再往真实系统走一步，还会多出几类你在线上必须考虑的问题：

    1. normalization 是否保留大小写、重音和 Unicode 变体
      这些决策会影响词表稀疏性，也会影响可逆性与下游语义信息。

    2. 特殊 token 如何冻结
      像 BOS、EOS、PAD、控制 token，通常不能被普通 merge 规则污染。

    3. encode/decode 是否严格 round-trip
      训练、评测、日志、标注对齐都很依赖可逆性；一旦回不去原文，debug 会非常痛苦。

    4. 吞吐是否足够高
      当训练数据是 TB 级时，tokenizer 自己就可能成为数据流水线瓶颈。

    一个极简训练例子¶

    假设训练语料只有：

    low
lower
lowest

    如果一开始按字符切，可能会多次看到：

    • l + o
    • lo + w
    • low + e

    那么 BPE 就会逐渐把高频子串组合成：

    • lo
    • low
    • er
    • est

    这件事的本质并不是“造新词”，而是：

    把重复出现的局部模式压缩成更短的记号，以节省后续模型处理长度。

    你也可以把它理解成：BPE 在做一个近似的“词表预算分配”。词表容量是有限的，于是它优先奖励那些全局复用次数最高的模式。

    一个更完整的 tokenizer pipeline¶

    很多初学者会把 tokenizer 理解成“一个 vocab 文件 + 一个 encode 函数”。实际工程里通常至少包含下面几层：

    1. Normalization¶

    典型操作包括：

    • Unicode 归一化；
    • 大小写折叠；
    • 去除或保留重音；
    • 特定字符替换。

    这一步本质是在决定：哪些输入差异应该被保留，哪些应该被视为同一种写法。

    例如，如果你做 aggressive normalization，那么词表会更紧凑，但你也可能丢掉大小写、拼写风格、甚至部分语言学信息。

    2. Pre-tokenization¶

    这一步会先把原始文本按某些规则切成较粗的片段，比如：

    • 空格边界；
    • 标点边界；
    • 数字边界；
    • 特殊 token 边界。

    它的重要性在于：它定义了 merge 可以发生的合法范围。

    如果你不先做这一层，BPE 可能会学到一些跨空格、跨标点的奇怪 merge；如果你切得太碎，又会损失本来可以压缩的高频结构。

    3. Subword Model¶

    这才轮到 BPE、WordPiece、Unigram 等真正的子词算法出场。

    它们共同要解决的问题是：

    • 如何让高频模式更短；
    • 如何让稀有词仍然能被表示；
    • 如何在词表大小与序列长度之间找平衡。

    4. Post-processing 与 Decoding¶

    真实模型通常还会有：

    • 加 BOS / EOS / SEP 等特殊 token；
    • 把内部空格标记还原；
    • 保证 token 到原文的映射可追踪。

    这些细节看起来像“收尾工作”，但对训练一致性、评测 reproducibility、以及错误分析都很关键。

    工程视角：Tokenizer 影响哪些下游成本¶

    你可以把 tokenizer 看成整个 LM 系统的第一个杠杆。它会影响：

    1. 训练成本¶

    同样一份数据，token 越少：

    • 每个 batch 的序列更短；
    • attention 更便宜；
    • 激活更省；
    • 通信量也可能下降。

    进一步说，token 数下降的收益不止一层：

    • 前向 attention 的 \(QK^T\) 和 softmax 开销会下降；
    • 反向传播也会随着序列长度缩短而减轻；
    • activation checkpointing、all-reduce、参数更新之外的很多辅助开销，也会因为 step 内张量更小而变得更轻。

    所以 tokenizer 压缩率提升，往往不是“节省一点点前处理成本”，而是在整个训练系统里持续复利。

    2. 推理成本¶

    尤其在长上下文推理里：

    • token 数越多，KV cache 越大；
    • decode 每步读取的历史越长；
    • TTFT 和吞吐都可能变差。

    在长上下文场景里，这个影响尤其直接。因为 decode 阶段通常越来越偏 memory-bound：

    • 你不是在做很大的新计算；
    • 你是在不断读取更长的历史 KV；
    • 所以“同样一段文本被切成多少 token”会直接决定缓存占用和每步访存压力。

    这也是为什么很多 serving 系统会非常关注不同语言、不同格式文本下的 token 膨胀问题。

    3. 多语言表现¶

    词表过小，某些语言会被切得很碎；
    词表过大，又会带来 embedding 成本、训练数据稀疏性和部署包体问题。

    多语言 tokenizer 里一个常见坑是：高资源语言把大部分 merge 容量“吃掉”，低资源语言则被迫退回到更碎的子词甚至 byte 级表示。结果就是：

    • 同一句语义相近的话，在不同语言里的 token 长度差很多；
    • 某些语言训练成本更高；
    • 某些语言在上下文窗口里更吃亏。

    所以多语言 tokenizer 设计，常常不只是“覆盖多少语言”，而是“不同语言是否被公平地分配了词表容量和压缩率”。

    4. 产品表现¶

    用户往往按 token 计费、按 token 限长。
    所以 tokenizer 其实还会影响：

    • 成本体验；
    • 长文输入能力；
    • 多语言产品一致性。

    在产品层面你甚至可以把 tokenizer 看成计费单位设计的一部分。因为用户并不感知“模型内部张量”，他们感知的是：

    • 为什么同样长度的中文和英文，计费 token 数差这么多；
    • 为什么某些格式化文本特别“费 token”；
    • 为什么一段代码、JSON、表格很容易把上下文挤满。

    这些现象的第一责任人，很多时候就是 tokenizer。

    怎么评价一个 tokenizer 好不好¶

    一个成熟的评价方式，不会只看“能不能 encode/decode”，而会至少看下面几件事：

    1. 压缩率¶

    最直接的问题是：平均每段文本会被切成多少 token。常见衡量方式包括：

    • bytes per token；
    • characters per token；
    • 不同语种、不同域上的平均 token 长度。

    压缩率越好，往往意味着同一窗口里能装更多语义内容。

    2. 覆盖率与鲁棒性¶

    要看它面对下面这些内容时是否稳定：

    • 生僻字符；
    • emoji；
    • 代码；
    • 多语言混写；
    • URL、表格、JSON、日志。

    如果 tokenizer 在这些输入上经常异常膨胀，那么系统成本和用户体验都会被拖垮。

    3. 可逆性与一致性¶

    encode 后 decode 能不能无损回去，训练和线上服务用的是不是同一套规则，这些都会直接影响：

    • 数据清洗一致性；
    • 评测可复现性；
    • 标注与偏移量对齐；
    • 故障排查效率。

    4. 训练分布匹配度¶

    一个 tokenizer 是否适合某个模型，不是抽象地问“它先进不先进”，而是问：

    • 它是不是在与你的训练数据相近的分布上学出来的；
    • 它是否覆盖你主要面对的语言和文本类型；
    • 它有没有对关键场景产生过高的 token 膨胀。

    脱离数据分布去谈 tokenizer，好坏往往是空的。

    面试里可以怎么讲这一讲¶

    如果面试官问：“为什么 tokenizer 很重要？”

    你可以用下面这版答法：

    Tokenizer 不只是文本预处理，它本质上在决定模型看到的离散单元是什么。这个选择会同时影响语义压缩率、序列长度、attention 复杂度、KV cache 大小和多语言表现。像 BPE 这种方法，本质上是在词表大小和序列长度之间做数据驱动的折中，所以它会对训练成本和推理成本都有直接影响。

    如果面试官问：“为什么要从头学语言模型，而不是直接调开源框架？”

    你可以答：

    因为很多决定模型效果的关键因素不在 API 层，而在 tokenizer、训练 loop、资源核算、并行策略和数据 pipeline 这些底层机制上。只有理解这些，才能判断一个结论到底是架构贡献、数据贡献，还是系统实现带来的假象。

    如果面试官继续追问：“你会怎么评价一个 tokenizer 设计是否合理？”

    你可以答：

    我会从四个维度看：第一是压缩率，也就是平均每段文本会被切成多少 token；第二是覆盖率和鲁棒性，尤其是多语言、代码、emoji、特殊符号这些边角输入；第三是可逆性和线上线下一致性；第四是它和训练分布是否匹配。因为 tokenizer 不是单独算法，而是整个训练和 serving 成本模型的一部分。

    本讲小结¶

    这一讲真正想建立的，不是 tokenizer 的算法细节，而是一种整体观：

    • 模型效果来自整条链路；
    • 资源效率必须和能力一起考虑；
    • tokenizer 是大模型系统里最前面的“压缩与表达接口”。

    后面所有主题——Transformer、GPU、并行、推理、数据、对齐——其实都能回到这套思路上继续展开。

    面试常见题目¶

    1. 为什么说 tokenizer 不是简单的文本预处理，而是模型系统设计的一部分？
    2. BPE、WordPiece、Unigram 的核心差别分别是什么？各自适合什么场景？
    3. 为什么很多现代 LLM 偏向 byte-level BPE，而不是纯字符级或纯词级 tokenizer？
    4. 如果把同一批训练语料换成更高压缩率的 tokenizer，训练和推理会分别受到什么影响？
    5. tokenizer 为什么会影响 KV cache 大小和长上下文吞吐？
    6. 一个多语言 tokenizer 设计不佳时，通常会在产品侧出现什么症状？
    7. 如果某种语言在你的系统里 token 膨胀特别严重，你会如何定位问题？
    8. tokenizer 的 normalization、pre-tokenization、subword merge 各自负责解决什么问题？
    9. 为什么说小模型实验里观察到的 tokenizer 优势，不一定能原样迁移到大模型？
    10. 什么时候你会考虑重新训练 tokenizer，而不是继续沿用现有 tokenizer？

    面试题答题提示¶

    1. 讲 tokenizer 重要性时，不要只停留在“切词更好”¶

    尽量把回答链路说完整：

    token 压缩率 -> 序列长度 -> attention / KV cache / 吞吐 -> 训练成本与推理成本

    只说“影响语义表达”太虚；把它和 FLOPs、显存、延迟、计费联系起来，回答会更像工程面试。

    2. 讲 BPE 时，重点不是背算法，而是讲清折中¶

    面试官通常更想听到你是否理解：

    • 为什么不能直接用词级；
    • 为什么也不能完全退回 byte 级；
    • BPE 到底在拿什么换什么。

    一句话版就是：用有限词表预算，优先压缩高频模式，同时保留对稀有字符串的可表示能力。

    3. 讲规模依赖时，要避免绝对化¶

    更好的说法是：

    • 某个结论可能在小规模实验成立；
    • 但随着参数量、数据量、上下文长度和系统瓶颈变化，收益会迁移；
    • 因此需要 scaling-aware 的验证方式。

    这类表述会比“规模大了就都不一样”更严谨。

    4. 讲多语言 tokenizer 时，最好落到公平性与成本¶

    你可以主动补一句：

    多语言 tokenizer 的问题不只是词表覆盖，而是不同语言是否获得接近的压缩率；否则某些语言会天然更贵、更短上下文、更难学。

    这通常会让回答明显高于“支持更多语言就行了”这种层次。
