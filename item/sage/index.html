    SAGE¶

    SAGE 是什么？¶

    SAGE 可以理解为一个面向科学探索的分布式复合型 AI 推理编排框架。它的目标不是只跑一个单点模型，而是把各类 AI 推理中间件、检索组件、预处理流程和执行资源编排在一起，形成端到端的任务链路，用来构建任务负载、模拟推理场景，并评测不同方案在效果和效率上的 trade-off。

    换句话说，SAGE 更像一个用于研究和验证“复杂 AI 推理系统应该怎么搭、怎么跑、怎么评估”的基础设施层，而不是一个单独的模型应用。

    这个项目要解决什么问题？¶

    SAGE 关注的不是“某个模型单点效果好不好”，而是更上层的问题：

    • 在复杂推理任务里，召回精度和最终推理效果是否一定正相关？
    • 在 RAG 场景下，海量数据进入 ANNS 后，检索链路应该怎样配合推理链路，才能真正提升最终答案质量？
    • 在新闻聚合、流式内容审阅这类高吞吐场景里，是不是必须把所有信息都一条条交给最终模型处理？
    • 能不能通过 embedding 预处理、分层筛选或中间件编排，降低最终模型的处理量，同时提高整体效率？

    这些问题本质上都不是单点模型问题，而是系统级推理编排问题。

    SAGE 的价值就在于：提供一个基于 Dataflow 的实验和编排框架，让研究者可以更方便地搭建流式仿真环境，系统性探索推理系统的边界、机制和优化空间。

    SAGE 的核心思路¶

    SAGE 基于 Dataflow 设计，把任务执行过程抽象成一条可以编排、组合和调度的流水线。

    它不是把一个复杂任务写成一大坨业务代码，而是把链路拆成：

    • 上层的逻辑算子与逻辑链路；
    • 下层的执行算子与物理链路；
    • 再由运行时去调度 CPU、GPU、内存和远程执行资源。

    这样做的好处是：

    • 向上提供统一的编程接口；
    • 向下屏蔽不同执行环境的复杂性；
    • 中间能插入更多可观测、可优化、可替换的执行环节。

    这类设计非常适合做复合型 AI 推理任务，因为任务本身往往不是一次模型调用，而是检索、预处理、推理、服务调用、后处理的组合链路。

    框架的逻辑分层¶

    按照目前的设计描述，SAGE 可以拆成下面几个核心层次。

    1. Dataflow API¶

    这一层向用户提供编排任务的一级抽象算子，例如：

    • source
    • map
    • sink
    • join
    • filter

    用户通过这些算子组合出一条 Pipeline 来表达自己的业务流程。

    从开发实现上看，这一层大致分成两部分：

    • function：由用户实现，承载具体业务逻辑；
    • operator：负责算子自身处理和链路通信，用户通常无感知。

    这一层的价值是把业务逻辑和运行时机制解耦，让用户更专注在“做什么”，而不是“怎么通信、怎么调度”。

    2. 逻辑图编译¶

    Dataflow API 组合出来的 pipeline 不会直接执行，而是先被解析成一个逻辑上的有向无环图（DAG）。

    这一层主要做的事是：

    • 识别不同 operator 的类型；
    • 分析 operator 的下游依赖；
    • 建立逻辑上的连接关系；
    • 形成可以进一步映射到运行时的逻辑图。

    这一步相当于把“用户写出来的流水线表达”变成“系统能理解和优化的执行蓝图”。

    3. 执行图执行¶

    在逻辑图之后，系统会继续把图解析成真正可运行的物理执行 pipeline。

    这一层主要包括：

    • 保存逻辑到执行的映射表；
    • 初始化各类执行算子；
    • 绑定上下游通信关系；
    • 把逻辑链路真正落到物理执行链路上。

    这也是 SAGE 真正把“抽象编排”变成“可执行系统”的关键一步。

    4. 算子通信后端¶

    SAGE 在通信执行层面区分本地和远程两种环境：

    • Local：基于 Python API 直接进行消息传递；
    • Remote：使用 Ray 做序列化、反序列化和远程任务执行。

    这一层的意义在于：

    • 本地调试时轻量直接；
    • 分布式运行时可扩展到远程资源；
    • 同一套编排逻辑可以在不同运行模式间切换。

    5. 中间件服务层¶

    除了算子之间的数据流，SAGE 还引入了服务化能力。

    你可以把它理解成：在执行 pipeline 的环境里，注册一个全局服务表，任何算子都能通过函数调用方式查找服务并发起请求。

    具体实现上，当前思路包括：

    • 一个监听队列（Source）
    • 一个返回队列（Sink）

    用户算子可以同步或异步调用服务，从而把“服务调用”也纳入统一 pipeline 编排体系里。

    这对复合推理场景很重要，因为现实里的推理链路常常不仅有模型算子，还会穿插：

    • 检索服务
    • 向量服务
    • 排序服务
    • 工具调用
    • 外部知识增强

    SAGE 的做法，相当于把这些中间件也纳入同一套执行框架里管理。

    从场景上怎么理解 SAGE¶

    场景一：RAG 的效果和检索精度真的总是一致吗？¶

    SAGE 关注的一个问题是：找回精度高，不一定意味着最终推理效果就一定更好。

    比如在 RAG 中：

    • 百万条信息进入 ANNS；
    • 检索链路做了某种筛选；
    • 最终模型拿到的是检索结果的一个子集。

    此时真正的问题不是“检索准不准”这么简单，而是：

    • 检索结果怎样被后续推理消费；
    • 哪些中间环节改变了信息分布；
    • 最终效果提升到底来自检索本身，还是来自链路编排方式。

    场景二：流式信息处理能不能更高效？¶

    比如新闻聚合、内容审阅这类任务，会持续有大量信息进入系统。

    这类场景下，并不一定适合把每条原始信息都交给最终大模型处理。更合理的方式可能是：

    • 先做 embedding 预处理；
    • 再做粗筛或聚类；
    • 最后只把高价值信息交给重推理链路。

    SAGE 想做的，就是把这种“分层处理、逐步缩小计算范围”的实验链路搭起来，并评估这种设计到底有没有真实收益。

    这项工作的核心价值¶

    如果从面试角度来讲，SAGE 的价值可以总结成下面几条：

    1. 不是做单点模型，而是在做推理系统基础设施¶

    它关注的是复杂 AI 推理任务如何被抽象成稳定、可编排、可扩展的执行链路。

    2. 强调效果与效率的统一评估¶

    不是只看精度，也不是只看吞吐，而是看完整系统在不同链路设计下的综合表现。

    3. 适合做流式和复合型推理实验¶

    对需要组合多个中间件、多个模型、多个处理阶段的任务，Dataflow 式设计天然更合适。

    4. 更偏系统设计而不是脚本拼装¶

    它不是“把几个库串起来”，而是把逻辑图、执行图、通信后端和服务抽象统一纳入框架设计。

    这个项目里体现出来的工程经验¶

    除了框架本身，原始笔记里还提到了几类非常适合面试说的工程经验。

    1. 大工程项目管理¶

    包括：

    • 建立 CI/CD 流程，验证基础服务架构是否正常工作；
    • 用 Pybind、PyTorch.ops 等方式完成 C++ 到 Python 的能力注册；
    • 用 pre-commit hook 强制执行单向依赖和提交流程规范；
    • 制定统一的 toml 配置，提升项目可复现性；
    • 做到代码和文档同步开发。

    这些内容的价值在于，它们体现的不是“会写代码”，而是会治理工程系统。

    2. Dataflow 编程范式的理解¶

    SAGE 背后反映的是一套面向流式数据处理的编程思想。

    这套思想更强调：

    • 把任务拆成稳定算子；
    • 把算子连接成可观测的 pipeline；
    • 把业务逻辑和运行时调度分开；
    • 让复杂系统具备可维护、可扩展、可实验的能力。

    3. 系统化做事的方法¶

    原始笔记里有一句很值得保留：

    即便有 vibe coding，工作结果仍然会有较大差距，关键在于代码标准化、鲁棒性测试和整体工程质量。

    这句话很像真正做过工程的人会有的判断。

    因为真正把东西做成系统，靠的不是一次性写通，而是：

    • 标准化
    • 测试
    • 约束
    • 可维护性
    • 长期演进能力

    如果在面试里怎么介绍 SAGE¶

    你可以用这版：

    SAGE 是一个面向复杂 AI 推理任务的 Dataflow 编排框架，核心不是做单点模型优化，而是把检索、预处理、推理、中间件服务和执行资源统一抽象成可编排的 pipeline，用来研究不同链路设计对效果和效率的影响。

    我在这个方向里关注的重点，一方面是如何把任务链路抽象成逻辑图、执行图和通信后端，另一方面是如何让系统既能支持本地轻量运行，也能支持基于 Ray 的远程分布式执行。

    这个项目的价值在于，它把复杂推理系统从“脚本堆叠”升级成“可组合、可评测、可演进”的基础设施能力。

    一句话总结¶

    SAGE 这段经历最适合包装成：我在做一个面向复杂 AI 推理任务的 Dataflow 编排框架，用统一的逻辑图、执行图和服务抽象来支撑流式实验、分布式执行和效果/效率协同优化。
