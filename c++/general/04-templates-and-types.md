# 模板与类型系统速查

## 核心定义（What & Why）

> **一句话总结**：模板是 C++ 在编译期生成类型安全通用代码的机制，它解决的是“同一套逻辑如何适配多种类型”这个问题，但也因此把很多复杂度提前到了编译期与错误诊断阶段。

## 要点

- 模板是“编译期代码生成”机制；错误往往在实例化时爆炸
- 理解：类型推导、特化 / 偏特化、SFINAE / 约束（C++20 Concepts）
- 真正难的往往不是“语法写出来”，而是推导、重载决议和错误信息能不能讲清楚

## 核心代码 / 操作演示（How）

### 一个最小模板函数

```cpp
template <typename T>
T add(T a, T b) {
	 return a + b;
}
```

### 转发引用与 `std::forward`

```cpp
#include <utility>

template <typename T>
void wrapper(T&& x) {
	 consume(std::forward<T>(x));
}
```

这里最关键的是：

- `T&&` 在模板推导场景里不总是右值引用；
- 它可能是**转发引用**；
- `std::forward<T>(x)` 的意义是保留实参原本的值类别。

## 对比表

| 概念 | 作用 | 优点 | 常见坑 |
|---|---|---|---|
| `auto` | 让编译器推导变量类型 | 简洁，减少冗长类型书写 | 会丢引用 / 顶层 `const` 细节 |
| `decltype` | 获取表达式的精确类型 | 保留表达式类型语义 | 规则细碎，括号影响结果 |
| `decltype(auto)` | 按 `decltype` 规则推导返回值/变量 | 能保留引用返回 | 容易无意中返回引用 |
| SFINAE / Concepts | 约束模板参与重载 | 更安全的泛型接口 | 错误信息和推导边界复杂 |
| 特化 / 偏特化 | 为特定类型定制行为 | 性能与语义可定制 | 容易让行为分叉过多 |

## 你需要能讲清楚的概念

- `typename` vs `class`（模板参数位置等价；依赖名需要 `typename`）
- `decltype` / `auto` / `decltype(auto)`
- 转发引用（`T&&` 在模板推导场景）与 `std::forward`
- Concepts 为什么比传统 SFINAE 更可读

## 💥 实战踩坑记录（Troubleshooting）

> 现象：模板代码能写出来，但一实例化报错就是几百行，看起来像编译器在念咒。

- **误判**：觉得“模板太玄学，靠背语法就行”。
- **根因**：很多时候是类型推导、重载决议或依赖名没有讲清；错误爆炸只是表象，根因往往是一个小小的推导边界。
- **解决动作**：
  - 先确认模板参数 `T` 最终被推导成了什么；
  - 再看值类别、引用折叠和 `const` 是否符合预期；
  - 如果接口本来就有前置要求，用 Concepts / `static_assert` 提前把约束写明白。
- **复盘**：模板最怕“看起来通用，实际上调用边界一团雾”；好的约束和错误信息本身就是设计的一部分。

> 常见异常：`std::initializer_list` 版本被意外匹配。

- 这通常是重载决议在“帮倒忙”；碰到 `{}` 初始化时，要特别警惕 `initializer_list` 重载比你想的更有吸引力。

## 常见模板坑

- 两阶段查找（two-phase lookup）
- ADL（参数依赖查找）
- `std::initializer_list` 参与重载决议导致的“意外匹配”

## 🎯 面试高频 Q&A

### 初级

1. 模板是什么？
	- 回答要点：模板是编译期生成泛型代码的机制，让同一份逻辑能适配多种类型而保持类型安全。
2. `typename` 和 `class` 在模板参数里有什么区别？
	- 回答要点：在模板参数位置几乎等价；但依赖名场景下通常必须写 `typename`。

### 中级

1. `auto`、`decltype`、`decltype(auto)` 有什么区别？
	- 回答要点：`auto` 更像普通推导，常忽略顶层 `const` / 引用；`decltype` 保留表达式语义；`decltype(auto)` 按 `decltype` 规则推导并常用于保留返回值引用语义。
2. 什么是完美转发？
	- 回答要点：在模板包装层里保留实参原本的左值 / 右值属性，通常借助转发引用和 `std::forward` 实现。

### 高级

1. 为什么 Concepts 往往比 SFINAE 更适合现代接口？
	- 回答要点：可读性更好、错误信息更直接、约束表达更贴近语义要求。
2. 两阶段查找为什么会让模板代码看起来“明明有这个名字却找不到”？
	- 回答要点：因为依赖名查找被推迟到实例化阶段，名字查找规则和普通函数体不同；这是模板代码容易“反直觉”的来源之一。
