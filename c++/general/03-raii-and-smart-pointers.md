# RAII 与智能指针

## 核心定义（What & Why）

> **一句话总结**：RAII 的核心是把资源生命周期绑到对象生命周期上，让资源在离开作用域时自动释放；智能指针则是这种思想在动态内存管理上的标准化落地，用来减少泄漏、悬垂和异常路径下的清理混乱。

## 要点

- RAII：资源的获取与释放绑定对象生命周期（构造获取、析构释放）
- 能不用裸 `new/delete` 就不用；优先值类型与标准库容器
- 默认优先 `std::unique_ptr`，共享生命周期才考虑 `std::shared_ptr`

## 核心代码 / 操作演示（How）

### 用 RAII 管理文件句柄

```cpp
#include <cstdio>
#include <memory>

int main() {
	 // 用 unique_ptr + 自定义 deleter 管理 FILE*
	 std::unique_ptr<FILE, decltype(&std::fclose)> fp(std::fopen("data.txt", "r"), &std::fclose);

	 if (!fp) {
		  return 1;
	 }

	 // 离开作用域时自动 fclose，无需手写清理分支
	 return 0;
}
```

### 用 `weak_ptr` 打破循环引用

```cpp
#include <memory>

struct Node {
	 std::shared_ptr<Node> next;
	 std::weak_ptr<Node> prev; // 观察，不拥有
};
```

这里的重点不是“语法会不会背”，而是要知道：

- `shared_ptr` 会增加拥有者计数；
- `weak_ptr` 不参与拥有权，所以不会把对象“拉住不放”。

## 对比表

| 工具 | 所有权语义 | 优点 | 代价 / 风险 | 更适合的场景 |
|---|---|---|---|---|
| 裸指针 | 不表达所有权 | 轻量、直观 | 容易泄漏、悬垂、重复释放 | 只做非拥有观察或与底层接口交互 |
| `unique_ptr` | 独占所有权 | 零额外引用计数、语义清晰 | 不能拷贝 | 默认首选的堆对象管理 |
| `shared_ptr` | 共享所有权 | 多处共同持有方便 | 控制块、原子计数、循环引用风险 | 需要共享生命周期 |
| `weak_ptr` | 非拥有观察 | 可打破循环引用 | 使用前需 `lock()` 检查有效性 | 观察者、缓存、反向引用 |

## `std::unique_ptr`

- 独占所有权；不可拷贝、可移动
- 支持自定义 deleter（管理 `FILE*`、socket、mutex handle 等）
- 默认首选，因为它的语义最明确、运行时成本也最低

## `std::shared_ptr` / `std::weak_ptr`

- `shared_ptr`：共享所有权（引用计数）
- `weak_ptr`：观察对象但不延长生命周期
- 循环引用通常要靠 `weak_ptr` 打破

## 何时用哪个

- 默认：`unique_ptr`
- 确实需要共享生命周期：`shared_ptr`
- 只观察不延长生命周期：`weak_ptr`
- 如果对象根本不需要堆分配，优先直接放栈上或交给容器管理

## 💥 实战踩坑记录（Troubleshooting）

> 现象：程序退出时没有崩，但内存持续涨，工具报告对象未释放。

- **误判**：一开始容易怀疑是某处忘了 `delete`，然后开始全局搜“哪里没释放”。
- **根因**：很多时候不是忘释放，而是两个 `shared_ptr` 互相持有，导致引用计数永远不归零。
- **解决动作**：
  - 先看对象关系图，找是否存在双向强引用；
  - 再把其中一侧改成 `weak_ptr`；
  - 最后复查该引用到底是“拥有”还是“观察”。
- **复盘**：智能指针不是“用了就自动安全”，关键是你有没有把所有权语义建模对。

> 常见异常：`double free` 或悬垂引用。

- 如果对象已经交给 `unique_ptr` / `shared_ptr` 管理，就不要再手动 `delete`；
- 如果把裸指针传出去，要先明确它只是借用还是会接管所有权。

## 🎯 面试高频 Q&A

### 初级

1. 什么是 RAII？
	- 回答要点：资源获取与对象构造绑定，资源释放与对象析构绑定；核心价值是异常安全和作用域自动清理。
2. 为什么默认优先 `unique_ptr`？
	- 回答要点：所有权语义最清晰、运行时开销更低、不会引入共享计数复杂度。

### 中级

1. `make_unique` / `make_shared` 为什么更推荐？
	- 回答要点：避免手写 `new`，异常安全更好；`make_shared` 还能把对象和控制块一起分配，减少一次分配开销。
2. `shared_ptr` 的代价是什么？
	- 回答要点：引用计数维护有成本；控制块带额外内存；原子操作会影响性能；还可能引入循环引用。

### 高级

1. `shared_ptr` 的线程安全边界在哪里？
	- 回答要点：不同 `shared_ptr` 实例对同一控制块的引用计数增加/减少通常是线程安全的，但被管理对象本身的读写并不会因此自动线程安全。
2. 什么时候不该用智能指针？
	- 回答要点：对象无需堆分配时；性能路径中所有权很明确时可直接用值语义或容器；智能指针不是替代所有设计思考的银弹。
