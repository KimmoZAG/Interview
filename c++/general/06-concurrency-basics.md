# 并发基础

## 核心定义（What & Why）

> **一句话总结**：并发编程解决的是“多个执行流如何安全协作访问共享状态”，它的难点不在于把线程开起来，而在于如何避免 data race、死锁、虚假唤醒以及那些最会在凌晨出现的时序 bug。

## 要点

- 区分：并发（concurrency） vs 并行（parallelism）
- 共享数据必须有同步：互斥量 / 原子 / 条件变量
- 正确性通常比“多开几个线程”更重要；先保证无 race，再谈吞吐优化

## 核心代码 / 操作演示（How）

### 用互斥保护共享计数器

```cpp
#include <mutex>
#include <thread>
#include <vector>

int counter = 0;
std::mutex m;

void worker() {
	 for (int i = 0; i < 1000; ++i) {
		  std::lock_guard<std::mutex> lock(m);
		  ++counter;
	 }
}
```

### 条件变量等待时要带谓词

```cpp
cv.wait(lock, [] { return ready; });
```

这行很短，但背后其实是在防：

- 虚假唤醒；
- 条件已经满足但通知发生在等待前；
- 只 `wait(lock)` 却忘了重新检查状态。

## 对比表

| 工具 | 更适合的场景 | 优点 | 主要风险 / 代价 |
|---|---|---|---|
| `std::mutex` + RAII 锁 | 复合共享状态保护 | 语义直接，易保证正确性 | 锁竞争、死锁、粒度难拿捏 |
| `std::atomic` | 简单计数器、标志位、无锁小状态 | 避免互斥开销 | 只适合简单状态，内存序难理解 |
| `std::condition_variable` | 线程协作、等待条件 | 不忙等，效率高 | 容易写出虚假唤醒 / 丢通知 bug |
| `std::thread` | 直接管理线程 | 控制明确 | join/detach 管理麻烦 |

## 线程与同步原语

- `std::thread`：线程启动与 `join` / `detach`
- `std::mutex` / `std::lock_guard` / `std::unique_lock`
- `std::condition_variable`
- `std::atomic`：只在需要时用；先掌握默认顺序一致，再考虑更细的内存序

## 💥 实战踩坑记录（Troubleshooting）

> 现象：程序偶发卡死，CPU 不高，也没有明确报错。

- **误判**：先怀疑第三方库、系统调度，或者“机器今天状态不好”。
- **根因**：经典死锁——两个线程以不同顺序获取两把锁，彼此互等。
- **解决动作**：
  - 先画锁顺序图，确认是否存在环；
  - 统一锁获取顺序，或用 `std::scoped_lock` 同时获取多把锁；
  - 再回看锁粒度是否过细导致顺序复杂失控。
- **复盘**：并发 bug 最怕“偶发”，但偶发不代表玄学，通常只是重现条件更挑剔。

> 常见异常：条件变量已经 `notify_one()` 了，等待线程还是卡住。

- 先检查等待是否使用了谓词版本 `cv.wait(lock, pred)`；
- 再检查共享状态的修改和通知是否在同一套同步规则下进行。

## 常见问题

- 数据竞争（data race）
- 死锁：锁顺序、锁粒度
- 虚假唤醒：`cv.wait(lock, pred)`

## 🎯 面试高频 Q&A

### 初级

1. 并发和并行有什么区别？
	- 回答要点：并发强调多个任务交错推进，并行强调多个任务同时执行；并发问题更关注共享状态和协调。
2. 为什么共享数据必须同步？
	- 回答要点：否则会出现 data race，程序行为未定义，结果可能偶发错误或崩溃。

### 中级

1. `lock_guard` 和 `unique_lock` 有什么区别？
	- 回答要点：`lock_guard` 更轻、更简单，适合作用域加锁；`unique_lock` 更灵活，可延迟加锁、解锁、配合条件变量。
2. 原子和互斥分别适用于什么场景？
	- 回答要点：原子适合简单独立状态；多个字段需要维护不变量时，互斥更可靠。

### 高级

1. 为什么说“atomic 不是 mutex 的性能升级版”？
	- 回答要点：atomic 只能安全处理单个原子状态或很小的无锁协议；复杂共享状态仍需要互斥来维护整体不变量。
2. 条件变量为什么一定要配谓词？
	- 回答要点：因为存在虚假唤醒和时序竞争；谓词保证线程醒来时重新检查共享条件，而不是盲目往下跑。
